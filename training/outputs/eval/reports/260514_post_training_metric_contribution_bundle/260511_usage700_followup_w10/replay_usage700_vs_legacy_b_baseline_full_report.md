# Replay Usage700 vs legacy_b / Baseline 全量评估报告

- 更新时间：2026-05-11
- 当前版本：`replay-usage700`，评估名 `260511_high_end_context_standard_w10` / `260511_high_end_context_hard_w10`
- 对照 legacy_b：sft_qwen25_7b_planner_1000_cp2_legacy_b/260511_high_end_context_*_w10，旧数据路线历史强对照，仅作 reference
- 对照 baseline：base_qwen25_7b/260511_high_end_context_*_w10，原始 Qwen2.5-7B-Instruct 基线
- 标准集：`training/data/planner/eval/records.jsonl`，200 条；困难集：`training/data/planner/eval_hard/records.jsonl`，300 条
- 推理设置：三组均为 temperature=0.2、workers=10；所有模型在 standard/hard 上均无调用失败、无重复缺口、无 finish_reason=length 截断。
- 规则口径：沿用当前 training/scripts/eval/eval_rule_metrics.py；meal_cost_scale_ok 只看 lunch/dinner 餐饮尺度，不把早餐纳入餐饮尺度硬判定。
- 命名说明：这里统一写“硬通过/软通过”，不使用训练阶段前缀命名

## 通过口径说明

硬通过部分只列 `hard_pass` 当前口径的组成项。预算合计、景点预算一致、餐饮预算一致、酒店覆盖晚数、用户预算约束等诊断项统一放在预算部分。

软通过链路：`软通过 = 硬通过 + 景点多样性 + 餐饮多样性 + 预算偏好贴合`；`重算预算软通过 = 硬通过 + 景点多样性 + 餐饮多样性 + 重算预算贴合`。

预算部分是诊断区，不等同于硬通过或软通过；其中 `预算偏好贴合` 和 `重算预算贴合` 同时也是软通过链路的预算项。

## 总结

当前 replay-usage700 在 500 条合并口径下，硬通过 96.6%，高于 legacy_b +3.0pp，高于 baseline +10.7pp。standard 硬通过为 95.5%，hard 硬通过为 97.3%。

软通过为 55.9%，重算预算软通过为 41.1%；软通过高于 legacy_b +8.0pp，高于 baseline +51.7pp。这两个指标更能反映硬约束之外的多样性和预算偏好收益。

预算诊断上，预算关系一致为 72.7%，高于 legacy_b +26.1pp，高于 baseline +66.9pp；重算预算贴合为 54.1%。分 split 的预算数值分布和变化摘要见下方表格。

hard split 的重算预算贴合仍需单独关注；它可能比合并视图更能暴露复杂样本上的预算目标贴合问题。

综合判断：当前候选在硬约束、软通过和合并重算预算贴合上整体领先；但困难集的重算预算贴合仍低于 legacy_b 1.5pp，下一步应继续补复杂预算目标贴合和选品样本。

## 读数建议

1. 读硬通过时只看 Hardpass 部分；预算合计类失败不要混进硬通过归因。
2. 读软通过时重点看 Softpass 部分里的景点多样性、餐饮多样性、预算偏好贴合和重算预算贴合。
3. Budget 部分用于解释预算问题，不要把每个预算诊断项都当成 hardpass/softpass 的组成项。
4. 对长期接近 0 或规则口径可疑的指标，优先判断它是不是字段口径或规则口径问题，避免误导选型。
5. 当前候选在硬约束、软通过和合并重算预算贴合上整体领先；但困难集的重算预算贴合仍低于 legacy_b 1.5pp，下一步应继续补复杂预算目标贴合和选品样本。
