# 旅行助手后训练指标贡献补充：每一轮改动到底值不值

这份补充文档不是为了再写一遍“我们做了很多优化”。我更想把它写成一份实验账本：哪一轮是真的涨了，哪一轮只是把一个问题换成另一个问题，哪一轮看起来很聪明但最后没有进入主线。

读这些数字时要先接受一个前提：这里很多改动不是严格的单变量实验。提示词、数据、采样温度、评测集难度、训练阶段都会互相影响。所以我会尽量用“贡献”和“代价”来讲，而不是把每个增量都说成绝对因果。

还有一个容易被忽略的点：前期和后期的评测口径不是一把尺子从头量到尾。早期 prompt 阶段，我们一边修协议，一边补评测脚本，预算、餐饮 grounding、人数关系这些指标都调整过；所以那部分数字更适合看同一批实验里的相对变化，不适合拿来和后期 SFT 指标硬比。到了 SFT 主线之后，standard / hard 两套评测集和指标脚本基本固定下来，这时候不同 LoRA、不同 checkpoint 之间的百分点对比才更有价值。

## 1. 我们到底在盯什么指标

旅行规划任务看起来是“生成一段行程”，但真正卡住模型的地方很具体：预算不能乱算，餐饮不能胡编，人数和房间不能对不上，景点和路线也不能只靠感觉。

| 指标 | 看什么 |
| --- | --- |
| `hardpass` | 结构、预算、餐饮、行程约束同时过线的比例。它最像最终交付质量。 |
| `softpass` | 在更宽松条件下，计划整体是否还算可用。 |
| `recomputed_budget_softpass` | 不相信模型自己写的预算，重新按结构化结果算一遍，看还能不能过。 |
| `budget_relationship_ok` | 预算是不是跟人数、天数、城市消费水平有关系，而不是拍脑袋。 |
| `budget_arithmetic_consistent` | 明细加总和总预算是否一致。 |
| `recomputed_budget_fit_ok` | 重算后的花费是否落在预算范围内。 |
| `meal_grounding_ok` | 餐厅是否能落到当天城市和合理场景里。 |
| `meal_cost_scale_ok` | 餐饮价格是否像真实世界，而不是一顿饭随手写几十或几千。 |

后面所有实验基本都绕着这几类指标打转。

## 2. Prompt 阶段：先把明显错的地方救回来

### 2.1 早期 prompt 的主要收益来自餐饮和约束

第一阶段非常朴素：把 prompt 从“像一个通用旅行助手”改成“你必须产出能被检查的结构化旅行方案”。这一步提升很大，但也暴露出一个问题：提示词越硬，模型越容易在某些局部指标上表现很好，同时在另一些地方开始僵。

| 版本 | `hardpass` | `softpass` | schema | 餐饮语义 | 餐饮落地 | 预算算术 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| `harder_w8` | 0.00 | 0.00 | 97.33 | 40.07 | 38.01 | 93.49 |
| `no_route_w8` | 0.67 | 0.00 | 99.33 | 38.26 | 35.23 | 96.64 |
| `food_bucket_no_route` | 53.72 | 26.01 | 98.67 | 60.81 | 60.81 | 94.93 |
| `no_route_meal_hotel` | 57.04 | 33.33 | 97.00 | 62.20 | 58.42 | 96.22 |
| `food_bucket_return_dinner` | 82.59 | 38.57 | 97.67 | 89.76 | 88.74 | 96.25 |

这里最明显的一跳是 `food_bucket_no_route`。餐饮从“随便列几个名字”变成“按早餐、午餐、晚餐的槽位生成”，`hardpass` 从几乎不可用直接到 53.72。后面的 `return_dinner` 又把晚餐回到城市、当天行程和消费预算里，餐饮落地涨到 88.74，`hardpass` 到 82.59。

这一段给我的教训很直接：早期别急着训练。先把 prompt 里那些会导致模型稳定犯错的自由度收掉，收益比想象中大。

### 2.2 上下文不是越多越好

后来我试过把上下文塞满，结果并没有更稳。旅行规划这种任务有点特别：模型需要足够的信息做约束，但信息太满时，它会在景点、餐厅、预算几条线之间来回抢注意力。

| 调整 | 正向变化 | 代价 |
| --- | --- | --- |
| `hard_full` -> `hard_compact` | 预算算术 60.54 -> 69.59，延迟 64.95s -> 60.85s，schema 98.00 -> 98.67 | 餐饮落地 97.28 -> 96.62，小幅回落 |
| `hard_compact` -> `hard_topk` | 延迟 60.85s -> 53.35s，schema 98.67 -> 99.33 | 餐饮落地 96.62 -> 94.97，景点落地 97.30 -> 95.97 |
| `hard_compact` -> `balanced_compact` | 餐饮多样性 6.76 -> 48.99 | 餐饮落地 96.62 -> 90.54，餐饮语义 97.30 -> 90.88 |

`hard_compact` 是比较舒服的点：它没有追求把所有候选都给模型，而是给足必要证据。`hard_topk` 更快，但落地率掉了；`balanced_compact` 的多样性很好看，不过它牺牲了我更在意的真实性。

所以这轮不是简单地“压缩上下文”，而是把上下文从百科式输入改成规划用输入。

### 2.3 预算 prompt：few-shot 分数高，但我不敢直接用

预算是最容易骗过人的指标。模型能写出很像样的预算说明，但如果按人数、天数、餐饮、酒店重新算，常常对不上。

| 版本 | `hardpass` | 餐饮落地 | 预算算术 | 预算偏好 | 备注 |
| --- | ---: | ---: | ---: | ---: | --- |
| `return_dinner_old` | 84.98 | 88.74 | 96.25 | 98.63 | 老版本基线 |
| `budget_hard` | 84.01 | 96.94 | 90.82 | 97.28 | 餐饮落地 +8.20pp，但预算算术 -5.43pp |
| `budget_order` | 84.30 | 94.88 | 91.81 | 97.61 | 稍微缓和一点 |
| `budget_fewshot` | 88.89 | 97.98 | 94.28 | 97.31 | 分数最好，但有示例数字污染 |
| `budget_symbolic` | 87.37 | 94.88 | 93.52 | 97.27 | 分数略低，泛化更干净 |

`budget_fewshot` 的结果一开始很诱人，`hardpass` 到 88.89。但检查样例后会发现，模型有时会借用 few-shot 里的数字模式。这个问题很烦，因为它不是马上爆炸的错误，而是会把数据生成阶段也带偏。

最后我更愿意保留 `budget_symbolic` 的思路：少给具体金额，多给计算关系。它的榜面分数不如 few-shot 漂亮，但更适合作为后面造数据和训练的底座。

### 2.4 账本实验和人数关系：看起来合理，实际没打通

我也试过让模型显式写预算账本。这个方向直觉上很对：先列账，再生成方案，最后对账。小样本结果却不行。

| 版本 | JSON | schema | 预算算术 | 景点预算 | 餐饮预算 | `hardpass` |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| `budget_ledger` | 5/5 | 5/5 | 2/5 | 0/5 | 0/5 | 0/5 |
| `budget_ledger_no_schema_numbers` | 5/5 | 0/5 | - | - | - | 0/5 |

账本最大的问题不是写不出来，而是写出来以后不一定约束后文。模型会很认真地列一份账，然后在餐厅或景点明细里又走丢。

人数和房间关系也类似。基线里 `budget_relationship_ok` 只有 6.76，酒店预算关系 39.53，景点预算人数关系 28.04。加了自检以后，小样本里预算语义从 5.26 到 20.00，酒店房间从 36.84 到 50.00，说明方向是对的，但还没有稳定到能靠 prompt 解决。

这也是后面转向 SFT 的原因之一：有些约束不是再补一句 prompt 就能解决的，它需要模型在数据里反复见到“应该怎么算”。

### 2.5 餐饮多样性：涨得快，也掉得快

餐饮多样性很容易做出漂亮数字。只要告诉模型“不要重复菜系、不要重复餐厅类型”，它就会马上听话。但旅行助手不是点评 App，餐饮最终还是要服务当天城市、路线、预算和用餐时间。

| 版本 | `hardpass` | `softpass` | 重算预算 | 餐饮多样性 | 餐饮尺度 | 餐饮落地 | 餐饮语义 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| `meal_diversity_symbolic` | 40.54 | 52.70 | 71.62 | 67.23 | 87.16 | 88.51 | 88.51 |
| `meal_count_limit_symbolic` | 41.69 | 54.58 | 73.56 | 69.49 | 86.10 | 87.46 | 88.47 |
| `meal_rotation_symbolic` | 44.22 | 55.78 | 70.07 | 70.41 | 89.46 | 90.48 | 90.14 |
| `meal_grounded_diversity_symbolic` | 41.61 | 50.34 | 69.80 | 64.09 | 90.60 | 92.28 | 90.94 |

`meal_rotation_symbolic` 最像一个局部最优：多样性高，餐饮落地也不差。但整体 `hardpass` 仍然不高，说明它解决的是餐饮子问题，不是完整旅行规划问题。

### 2.6 self-check：它能修一部分，也会弄坏一部分

自检 prompt 的结果很典型。它确实让模型更注意“不要重复”和“人数关系”，但预算算术和餐饮尺度掉得比较明显。

| 指标 | 基线 | self-check | 变化 |
| --- | ---: | ---: | ---: |
| schema | 98.7 | 97.7 | -1.0pp |
| 预算关系 | 6.8 | 8.2 | +1.4pp |
| 酒店预算关系 | 39.5 | 42.3 | +2.8pp |
| 景点人数预算关系 | 28.0 | 30.0 | +2.0pp |
| 餐饮价格尺度 | 87.2 | 79.5 | -7.6pp |
| 预算算术 | 69.6 | 56.3 | -13.3pp |
| 餐饮落地 | 96.6 | 91.8 | -4.8pp |
| 餐饮多样性 | 6.8 | 22.2 | +15.4pp |
| 同日午晚餐不重复 | 55.1 | 67.9 | +12.9pp |

这轮之后我对 self-check 的看法变谨慎了。它不是没用，但它很容易把模型注意力从“算对”转移到“检查声明看起来完整”。如果没有更强的执行约束，自检会变成一种额外作文。

### 2.7 最后进入数据生产的 prompt，不一定是单项最高分

最终 prompt 组合不是选某个指标最高的版本，而是选“比较均衡、坏味道少、适合继续训练”的版本。

| 版本 | schema | 预算算术 | 酒店预算 | 餐饮多样性 | 餐饮落地 | 餐饮落地率 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| baseline | 99.00 | 50.51 | 36.36 | 71.38 | 56.23 | 83.28 |
| A | 98.67 | 55.07 | 35.81 | 70.27 | 56.08 | 82.90 |
| D | 98.67 | 49.66 | 36.49 | 73.65 | 58.11 | 84.09 |
| A_C | 99.33 | 58.72 | 40.27 | 66.11 | 52.68 | 82.95 |
| A_D | 100.00 | 49.67 | 36.00 | 71.67 | 59.33 | 83.65 |
| A_C_D | 99.67 | 54.18 | 38.13 | 68.90 | 56.19 | 83.21 |

`A_C` 的餐饮落地不是最高，但预算算术和酒店预算关系更好，schema 也稳。这个选择有点反直觉，不过后训练里经常会遇到这种事：你不是在选一个 demo 最好看的 prompt，而是在选一个能产出可学数据的老师。

## 3. SFT 阶段：真正把“会不会做”写进模型

Prompt 阶段救的是行为边界，SFT 阶段开始改变模型的默认解法。最明显的变化是：模型不只是知道要写预算，而是更愿意按人数、天数、餐饮和住宿去组织答案。

从这里开始，评测口径就稳定很多了。后面这些 SFT 对比基本都在同一套 standard / hard 数据和同一套规则脚本下跑，所以可以更放心地看 `+5pp`、`-3pp` 这种变化。

### 3.1 第一轮干净数据：收益最大，也最清楚

第一条主线是用干净 SFT 数据训练 LoRA。这里的提升非常扎实，尤其是 `softpass`、预算关系和餐饮尺度。

| 评测集 | 指标 | base | LoRA SFT | 变化 |
| --- | --- | ---: | ---: | ---: |
| standard | `hardpass` | 89.90 | 96.46 | +6.56pp |
| standard | `softpass` | 5.56 | 61.11 | +55.55pp |
| standard | 重算预算 `softpass` | 13.13 | 45.45 | +32.32pp |
| standard | 预算关系 | 6.06 | 56.57 | +50.51pp |
| standard | 餐饮尺度 | 35.35 | 73.23 | +37.88pp |
| standard | 预算算术 | 53.03 | 76.26 | +23.23pp |
| standard | 餐饮落地 | 94.95 | 98.48 | +3.53pp |
| hard | `hardpass` | 83.28 | 91.97 | +8.69pp |
| hard | `softpass` | 3.34 | 46.15 | +42.81pp |
| hard | 重算预算 `softpass` | 3.34 | 35.12 | +31.78pp |
| hard | 预算关系 | 5.69 | 45.15 | +39.46pp |
| hard | 餐饮尺度 | 38.80 | 68.56 | +29.76pp |
| hard | 预算算术 | 52.51 | 65.89 | +13.38pp |
| hard | 餐饮落地 | 92.98 | 97.32 | +4.34pp |

这一轮最重要的不是 `hardpass` 涨了多少，而是 `softpass` 从个位数跳到 40 到 60 这个区间。它说明模型终于不只是在格式上过关，而是在预算和餐饮这些子任务上开始像一个规划器。

训练参数上，这轮我没有追求很大的 LoRA 容量，学习率也控制在比较保守的范围。原因很简单：数据还在验证期，先确认模型能不能学到“预算关系”和“餐饮尺度”，不要一上来就把基座风格冲坏。

### 3.2 usage700 replay：不是所有指标都涨，但方向对了

第二阶段加了 usage700 replay。它的作用更像“把真实使用中的难题再喂一遍”，不是单纯追求标准集涨分。

| 评测集 | `hardpass` | `softpass` | 重算预算 `softpass` | 重算预算 fit | 预算偏好 | 预算关系 | 餐饮尺度 | 餐饮落地 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| standard 变化 | -0.96 | -0.11 | +5.05 | +4.93 | +1.73 | +24.43 | +9.77 | -0.48 |
| hard 变化 | +5.35 | +6.36 | -0.34 | -5.02 | 0.00 | +22.07 | +11.04 | +1.68 |

这组数很有意思：standard 的 `hardpass` 小跌，但 hard 集明显涨。也就是说，它不是让简单题更漂亮，而是让模型在难题上更抗打。预算关系和餐饮尺度继续涨，说明 replay 的样本确实打到了痛点。

我会把这一轮理解成“鲁棒性补课”。它不适合拿单个榜面分数宣传，但对真实旅行助手更有价值。

### 3.3 patch700-only：单独补丁数据会偏科

我也单独试过 patch700-only。结果很适合作为反例。

| 评测集 | `hardpass` | `softpass` | 重算预算 `softpass` | 预算偏好 | 预算关系 | 餐饮尺度 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| standard 变化 | +1.04 | -26.61 | -14.95 | -32.27 | -3.07 | -12.73 |
| hard 变化 | +6.36 | -10.36 | -5.35 | -14.72 | +12.71 | -2.34 |

它能让某些 hard case 过线，但会把 `softpass` 和预算偏好打下去。这个现象很像只拿错题本训练：模型会变得很会处理错题本里的模式，却忘了普通题该怎么自然完成。

所以 patch 数据最后不能单独上主线，必须和 replay、干净样本混起来。

### 3.4 Best-of-N 600 replay：小幅但均衡的补强

Best-of-N 600 replay 的贡献不夸张，但比较健康。

| 指标 | 变化 |
| --- | ---: |
| `hardpass` | +2.0pp |
| 预算关系 | +6.8pp |
| 餐饮尺度 | +2.2pp |
| 预算算术 | +3.6pp |
| 景点多样性 | +1.4pp |
| 餐饮多样性 | +3.0pp |
| `softpass` | 0.0pp |
| 重算预算 `softpass` | -0.6pp |
| 预算偏好 | -4.6pp |
| 重算预算 fit | -2.8pp |

拆开看，standard 集 `hardpass` +3.5、`softpass` +4.5、重算预算 +3.0；hard 集 `hardpass` +1.0、预算关系 +9.4，但 `softpass` -3.0、预算偏好 -6.4、重算预算 fit -3.0。

这一轮我会给它一个比较现实的评价：它提升了难例覆盖，但没有解决重算预算 fit 的根问题。它适合继续作为训练材料，不适合被当成终点。

### 3.5 Best-of-N 1200：继续加数据后，收益开始变钝

后面把 Best-of-N 扩到 1200，并看了多个 checkpoint。standard 集上，ckpt104 的 `softpass` 和预算关系最好，ckpt156 的重算预算和餐饮落地更好，final 又有一点回落。

| 版本 | schema | `hardpass` | `softpass` | 重算预算 `softpass` | 重算预算 fit | 预算偏好 | 预算关系 | 餐饮尺度 | 预算算术 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 600 final | 99.00 | 65.50 | 53.50 | 84.00 | 77.00 | 59.50 | 70.50 | 86.00 | 82.50 |
| 1200 ckpt52 | 98.99 | 64.14 | 52.02 | 80.81 | 76.26 | 59.60 | 71.21 | 83.84 | 79.29 |
| 1200 ckpt104 | 98.99 | 69.85 | 54.27 | 80.40 | 80.90 | 59.80 | 64.82 | 83.92 | 78.39 |
| 1200 ckpt156 | 97.99 | 64.32 | 54.27 | 82.91 | 78.39 | 62.31 | 67.84 | 84.42 | 80.90 |
| 1200 final | 98.99 | 61.81 | 52.26 | 83.42 | 74.87 | 60.30 | 64.32 | 85.43 | 82.41 |

这里我不太愿意说“1200 一定比 600 好”。更准确的说法是：1200 让模型见到了更多候选解，但收益开始分散，checkpoint 选择变得重要。ckpt104 的 `hardpass` 到 69.85，是这组里最好看的；final 的预算算术不错，但 `hardpass` 反而回落到 61.81。

这说明后训练后半段不能只看最终 checkpoint。尤其是 LoRA SFT，多训一点不一定更好，评测曲线要留着，最好按目标指标选 checkpoint。

### 3.6 rerank：主增益开始从训练转向推理时选择

最后一轮我们把 `old600final`、`ckpt104`、`final1200` 都接到同一套多温度候选 + 规则 rerank 流程里。这个结果基本给 SFT 阶段画了句号：继续训 checkpoint 的收益，已经不如把候选生成出来再选得更稳。

500 条合并口径下，rerank 后结果如下：

| 版本 | JSON | `hardpass` | `softpass` | 重算预算 `softpass` | 预算合计/算术 | 预算偏好 | 预算关系 | 餐饮尺度 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| ckpt104 + rerank | 100.0 | 98.0 | 65.6 | 54.6 | 81.2 | 77.0 | 86.4 | 88.8 |
| final1200 + rerank | 100.0 | 98.2 | 66.8 | 54.6 | 78.0 | 78.4 | 85.0 | 88.0 |
| old600final + rerank | 100.0 | 98.2 | 66.2 | 59.2 | 78.4 | 75.4 | 87.0 | 89.4 |

这个表里最重要的不是谁第一，而是 rerank 把三个版本都拉到了更高平台。`final1200 + rerank` 的 `softpass` 最好，适合作为默认主线；`old600final + rerank` 的重算预算 `softpass` 最强，适合作为预算保守对照；`ckpt104 + rerank` 的预算算术最好，但整体不再是唯一候选。

所以 SFT 阶段的最终判断是：训练已经把模型推到可用区间，后续主要收益来自候选选择、预算重算和偏好判断。继续追加 SFT 可以做，但不再是当前性价比最高的方向。

## 4. 按阶段看，贡献大概长这样

如果把所有实验压成一句话，我会这样拆：

| 阶段 | 主要贡献 | 代价或风险 |
| --- | --- | --- |
| 早期 prompt | 把 `hardpass` 从几乎不可用拉到 80+，主要靠餐饮槽位和行程约束 | 很容易为了局部指标牺牲自然性 |
| 上下文压缩 | 降低延迟，预算算术更稳 | top-k 过窄会伤到落地率 |
| symbolic 预算 prompt | 减少 few-shot 数字污染，让数据更适合训练 | 榜面分数不一定最高 |
| self-check | 提升餐饮多样性和部分人数关系 | 预算算术、餐饮尺度会掉 |
| clean SFT | `softpass`、预算关系、餐饮尺度大幅提升 | 需要干净数据，脏样本会放大 |
| usage replay | hard 集更稳，预算关系继续涨 | standard 集部分指标小幅回落 |
| patch-only | 能打部分难例 | 偏科明显，不能单独作为主线 |
| Best-of-N replay | 难例覆盖更好，指标更均衡 | 数据继续加大后收益变钝，需要选 checkpoint |
| rerank | 多温度候选后选择更稳，`softpass` 和重算预算明显提升 | 增加推理成本，需要工程化延迟和兜底 |

我现在回看这条线，会觉得最值的三步是：

1. 用 prompt 先修餐饮槽位和预算表达，把数据生产的下限抬起来。
2. 用干净 SFT 让模型真的学会预算关系和餐饮价格尺度。
3. 用 replay 和 Best-of-N 去补 hard case，但保持评测选择 checkpoint，不迷信最终轮。

最不该高估的是 few-shot 预算和 self-check。它们确实能涨一些数，但都容易带来隐藏副作用。前者会污染数字分布，后者会让模型把注意力花在“看起来检查过”而不是真的算对。

## 5. 给开源读者的建议

如果你也要做一个旅行助手的后训练，我不建议一上来就跑大训练。更稳的顺序是：

1. 先写一个能失败的评测集，让预算、餐饮、人数、城市落地这些问题真的暴露出来。
2. 用 prompt 修最明显的结构错误，尤其是餐饮槽位、每日行程、预算字段和可重算字段。
3. 用强模型生成候选数据，但一定要审计。不要只看回答顺不顺，要看它能不能被脚本重新计算。
4. 第一轮 LoRA SFT 用干净数据，小步训练，看 `softpass`、预算关系、餐饮尺度有没有一起涨。
5. 再把真实失败样本和 Best-of-N 样本 replay 进去，专门补 hard case。
6. 每个 checkpoint 都评测。最终模型不一定来自最后一步，很多时候来自中间某个更平衡的点。

这条路并不神秘，更多是耐心活。旅行助手的难点不是写出一段漂亮行程，而是让每个数字、每顿饭、每个城市、每个约束都能互相对上。后训练真正带来的提升，也基本都发生在这些不起眼的地方。
