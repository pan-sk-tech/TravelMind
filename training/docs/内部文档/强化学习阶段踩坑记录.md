# 旅行规划模型强化学习阶段踩坑记录

## 背景

这一阶段的目标不是让模型“更像会说话”，而是让模型更像一个可靠的旅行规划器：选得对、预算合适、体验不单薄、输出能直接被系统消费。

从 SFT final 到 fd600 direct402，再到 ps2400 clean、closing budget clean 和最终 rerank，我们确认了三件事：

- DPO 信号是有效的，`planner_soft_pass` 从 SFT final 的 35.2% 提升到 fd600 ckpt-25 的 44.7%，再到 260519 ckpt-138 单生成的 71.5%。
- 最后一炉 DPO 不一定继续提升单生成，但可能改变候选分布；260521 ckpt-64 单生成一般，配合 rerank n4 后 combined `planner_soft_pass` 达到 80.6%。
- 训练收益非常依赖评估指标、数据干净程度和推理口径；指标、数据或 single/rerank 口径混了以后，模型选择会被误导。

所以后续强化学习阶段的核心原则是：先把目标指标、数据来源、筛选规则、评估口径和推理方式固定住，再谈继续训。

## 坑 1：只看 hard pass 会误判模型质量

`hard_pass` 很重要，但它只能说明计划基本可解析、字段正确、POI grounding 正确、日期城市没有错。它不能说明这个旅行计划体验好。

典型问题：

- 行程合法，但景点重复。
- 餐饮合法，但每天都像随便填的。
- 预算没有越界，但消费尺度不符合用户档位。
- 总预算看起来合理，但实际 POI 组合重算后不匹配。

后续策略：

- `hard_pass` 做底线，不做主优化目标。
- 主目标改成 `planner_soft_pass`。
- 每次模型选择都同时看 standard / hard split，不能只看 combined。

## 坑 2：旧 soft 指标不够贴近真实旅行体验

之前的 `dpo_soft_pass` 是：

```text
dpo_soft_pass =
  sft_hard_pass
  && attraction_diversity_ok
  && meal_diversity_ok
  && budget_preference_aligned
```

这个指标有价值，但 `budget_preference_aligned` 比较粗，容易让模型学会“预算描述像正确答案”，而不一定真的选出预算合适的酒店、餐饮、景点。

后来加入 `dpo_soft_recomputed_budget_pass`：

```text
dpo_soft_recomputed_budget_pass =
  sft_hard_pass
  && attraction_diversity_ok
  && meal_diversity_ok
  && recomputed_budget_fit_ok
```

这比旧 soft 更好，但仍然缺两个关键项：

- 用户硬预算是否被重算预算突破。
- 餐饮价格尺度是否符合预算档位。

后续策略：

```text
planner_soft_pass =
  sft_hard_pass
  && attraction_diversity_ok
  && meal_diversity_ok
  && meal_cost_scale_ok
  && recomputed_budget_user_constraint_ok
  && recomputed_budget_fit_ok
```

把 `planner_soft_pass` 定为下一阶段主目标。

## 坑 3：预算指标之间可能互相冲突

我们之前遇到过两个方向的张力：

- `hard_or_user_budget`：强调不能超过用户硬预算。
- `below_target_min` / budget fit：强调不能低于目标预算档位太多。

这两者不是同一个问题。一个是底线，一个是体验匹配。

如果只优化“不超预算”，模型可能变得过分保守，高预算用户也安排得很便宜。
如果只优化“贴近预算档位”，模型可能冒着超过用户硬预算的风险。

后续策略：

- `recomputed_budget_user_constraint_ok` 是底线。
- `recomputed_budget_fit_ok` 是偏好匹配。
- 训练数据里要明确区分这两类失败，不要混成一个模糊的 budget bad。

## 坑 4：餐饮价格尺度不能简单粗暴

高预算档位不能长期吃太便宜的午餐/晚餐，这是对的。但旅行里确实会有小吃、夜市、老街、美食街、早点、面馆、粉、包子、烧饼、糕点、甜品等本地特色。

如果规则过硬，会误伤真实体验：

- 高预算用户也可能想吃当地小吃。
- 夜市/老街/美食街可能是体验点，不应被简单判成低质。
- 早餐和甜品不应该和正餐用同一价格尺度判断。

后续策略：

- `meal_cost_scale_ok` 只重点约束高预算档位的午餐/晚餐。
- 本地特色小吃类词条要豁免或降权。
- 不把 comfortable 档位过度严格化，重点看 luxury / high budget 的明显低价正餐问题。

## 坑 5：直接拿 eval 失败 case 训练会数据泄漏

失败 case 很适合做诊断，但不能直接拿 eval 失败样本生成 DPO 数据继续训练，否则评估会被污染。

后续策略：

- 用 eval 失败 case 总结失败模式。
- 用 train-only prompts / contexts 生成同类压力样本。
- 保留独立 eval records 不参与训练。
- 每次新数据都记录来源、分布、筛选规则和随机种子。

## 坑 6：DPO pair 不是越多越好，信号要干净

如果 chosen 和 rejected 同时差很多东西，模型不知道到底该学哪一点。

坏 pair 示例：

- chosen hard pass，rejected JSON 都错了。这种 pair 只会教格式，不会教 planner soft。
- chosen 和 rejected 同时差预算、餐饮、景点、天气，信号太混。
- rejected 太烂，DPO 学不到细粒度偏好。

后续策略：

- chosen：hard pass，并尽量 planner soft pass。
- rejected：最好 hard pass，但在一个明确 planner soft 维度失败。
- 每个 pair 标注 primary failure，例如 `budget_fit`、`meal_scale`、`meal_diversity`、`attraction_diversity`。
- 数据分布按失败类型配比，不要只堆 budget case。

## 坑 7：prompt 已经接近写到头，继续调 prompt 收益有限

我们做过多轮 prompt 消融。当前问题更多像模型内化能力不足，而不是 prompt 没写清楚。

继续往 prompt 里塞规则的风险：

- 上下文更长，截断风险更大。
- 模型更容易机械复述规则。
- 服务成本和延迟变差。
- 对结构性选择能力提升有限。

后续策略：

- prompt 保持稳定，避免频繁改评估口径。
- 把主要精力放在数据构造、pair 筛选和训练目标上。
- prompt 只做必要的 bug fix，不作为主优化手段。

## 坑 8：max tokens / 截断会污染数据质量

强模型或本地模型生成时，如果 max tokens 太小，会出现 length 截断。截断样本进入 DPO 会污染训练，模型会学习到不完整输出或坏结构。

后续策略：

- 数据生成阶段记录 `finish_reason`。
- length / truncated 的样本默认不进入训练。
- 本地/强模型生成都要统一 max_tokens 和输出完整性检查。
- 长上下文任务优先保证完整输出，再考虑并发。

## 坑 9：外部强模型 endpoint 必须显式记录

这一轮曾经出现过“以为在打 mimo，实际打到别的强模型服务”的风险。模型来源混了以后，数据不是不能用，但后续分析会麻烦。

后续策略：

- 每批生成数据记录 provider、model name、base url、concurrency、max_tokens。
- 输出目录名里带 provider。
- 混源数据可以用，但必须在 manifest 里写清楚。
- 后续筛选时按 provider 做统计，避免误判某个模型的生成质量。

## 坑 10：vLLM 并发不是拍脑袋，按服务推荐值来

一开始容易用固定并发，比如 3、10、50、100。但真实吞吐取决于：

- vLLM 当前 KV cache 能承载多少并发。
- 单条输出长度。
- tensor parallel / GPU 数。
- max model len 和 max tokens。
- 是否一张卡评一个版本，还是多卡并行一个版本。

后续策略：

- 启动 vLLM 后先看 `Maximum concurrency`。
- 单模型单卡评估时，workers 大致按推荐并发取整。
- 多 checkpoint sweep 时，一张卡一个服务，一个版本一个端口，避免互相抢显存。
- 评估完必须自动停止服务并释放显存。

## 坑 11：中间 checkpoint 必须按 0.5 epoch 存

这轮结果说明训练曲线不是单调的：

- ckpt-25 综合最好。
- ckpt-30 某些指标更强。
- ckpt-35 明显回落。
- ckpt-40 又有部分恢复，但不如 ckpt-25 均衡。

如果只保存最终 checkpoint，很可能错过最佳模型。

后续策略：

- 每 0.5 epoch 保存一个可评估 adapter。
- 训练完成后自动 sweep 所有中间 adapter。
- 不只看 train loss，也看 planner soft / hard split / budget fit。
- 最终模型不默认取最后一步，按 eval 选择。

## 坑 12：`save_only_model: true` 不能当完整断点续训

本轮 `checkpoint-25` 有完整 LoRA adapter，可以作为下一轮初始化。但因为配置里是：

```yaml
save_only_model: true
```

它不是完整 optimizer / scheduler / DeepSpeed 状态的无缝 resume。

后续策略：

- 如果要严格断点续训，需要保存完整训练状态。
- 如果只是阶段二训练，推荐用 `adapter_name_or_path` 加载 ckpt-25，开新 output_dir。
- 从 ckpt-25 继续做新数据训练时，降低学习率，例如 `1e-6` 到 `1.5e-6`。
- 新数据训练 1.5 到 2 epoch 起步，每 0.5 epoch 存 adapter。

## 坑 13：继续训练不能只喂新数据

如果下一批只用新造强信号数据，可能会把上一轮已经学好的能力冲掉，尤其是 meal scale 和预算约束。

后续策略：

- 新强信号 clean data 占 70%-80%。
- 上一批 direct402 高质量数据保留 20%-30%。
- 加少量 hard-valid 普通样本做稳定器。
- 每批数据都跑分布报告：失败类型、预算档位、城市、天数、party size、comfortable/luxury 占比。

## 坑 14：评估上下文改了，历史结果不能硬混

我们之前重做过 comfort hotel / budget 相关 eval context。不同 records 或不同上下文下的指标不能直接放在同一主表里比较。

后续策略：

- 主对比表必须使用同一套 `records.jsonl`。
- 如果 records 不同，只能放在附录或单独说明。
- 报告里写清楚 standard/hard records 路径。
- 每次改规则后，至少重算关键 baseline。

## 坑 15：GPU 和服务生命周期要自动化

训练、vLLM、评估混在一起时，很容易出现显存没释放、服务还在、端口占用、日志不知道在哪的问题。

后续策略：

- 训练完成自动启动 vLLM 评估。
- 评估完成自动 stop vLLM。
- 每个服务写 pid、port、gpu、model path、log path。
- 清理显存时只杀自己启动的 pid，不误伤别人任务。
- 不启动无意义的占卡任务；GPU 使用要和真实训练/评估任务绑定。

## 坑 16：单生成最优和 rerank 最优不是一回事

260519 `ps2400clean_plus_direct402 checkpoint-138` 是目前单生成最稳的点，combined `planner_soft_pass` 到 71.5%，hard pass 保持 98.4%。

但最终展示版本不是它，而是 260521 `checkpoint-64` 配合 rerank n4。`checkpoint-64` 单生成并不更好，combined `planner_soft_pass` 只有 69.7%；可它在多温度候选里提供了更好的搜索空间，rerank n4 后 combined `planner_soft_pass` 到 80.6%，hard split planner soft 到 77.0%。

后续策略：

- 报告里明确区分 single generation 和 rerank n4，不把两类指标混在同一行里硬比。
- 单生成模型选择优先看 260519 ckpt-138。
- 产品展示或 demo 如果允许多候选 rerank，优先用 260521 ckpt-64 rerank n4。
- 后续训练不要因为 ckpt-64 单生成回落就直接判定 closing 数据没价值，要同时看候选分布和 rerank 后指标。

## 坑 17：closing 数据没涨单生成，不等于没价值

260520/260521 的 `closing_budget_clean` 主要想补预算过高、预算过低、边界和重复问题。单生成评估里它没有继续推高 planner soft：260520 ckpt-66 从 71.5% 回到 70.1%，260521 ckpt-64 又到 69.7%。

如果只看单生成，这两炉应该回退到 ckpt-138。但 rerank 后，ckpt-64 在 hard split 上比 ckpt-138 更强，combined `recomputed_budget_soft`、`recomputed_budget_fit` 也更好。

后续策略：

- closing 类数据要同时评估单生成、N-best 候选质量和 rerank 后结果。
- 对“修预算/修重复”的数据，不只看 overall planner soft，也看 hard split、budget fit、recomputed budget soft。
- 如果目标是提升单生成，closing 数据需要更谨慎控制 LR、epoch 和 replay 配比。
- 如果目标是提升最终系统表现，closing 数据可以作为改善候选池的工具。

## 坑 18：DPO loss 不能跨数据批次横比

fd600 direct402 的 pair 信号更强，loss 下降比较顺；closing budget clean 的 pair 更接近、更难，loss 在 0.4 到 0.55 并不自动说明训练坏了。

真正要看的不是“这炉 loss 是否比上一炉低”，而是：同一批数据上 loss 是否稳定下降，reward accuracy 是否合理，最终 eval 是否提升目标指标。

后续策略：

- 不跨数据集直接比较 DPO loss 数值。
- closing pair 更难时，允许 loss 高一些，但必须看 checkpoint sweep。
- 训练日志只能解释训练过程，不能替代 rule eval。
- 选择模型时以 `planner_soft_pass`、hard split、budget fit、recomputed budget soft 为准。

## 坑 19：预算收尾数据必须先做 signature anti-leak

closing budget clean 这轮的一个关键保护是先按冻结 eval 请求签名过滤，再选训练 pair：`frozen eval signature count = 497`，`selected eval signature overlap = 0`。

这个步骤不能省。预算 bad case 很容易从 eval 里总结出来，如果不做 signature 过滤，后续分数会看起来漂亮，但实际上已经污染了评估。

后续策略：

- pair selection 前先建立 frozen eval signature set。
- train pair 选完后强制报告 selected eval overlap。
- 尽量 one pair per request signature，避免某类 prompt 被重复记忆。
- review samples 里保留 budget category、chosen/rejected 指标和来源说明。

## 坑 20：直接把 planner soft 当唯一 RL 奖励太稀疏

`planner_soft_pass` 适合作为主评估指标，但如果后面做 GRPO / RLOO，不能只给一个 0/1 奖励。TripPlan 很长，失败原因可能是 schema、grounding、预算、餐饮尺度、重复、截断中的任意一种；只用 0/1 会让 credit assignment 太粗。

后续策略：

- DPO 收益还没耗尽前，继续优先做高质量 DPO pair。
- 下一步 RL-style 实验先做 Best-of-N + 规则奖励，复用现有 evaluator。
- 真要在线 RL，优先小规模 GRPO / RLOO，不要一上来做 PPO。
- reward 用 shaped reward：JSON/schema、hard pass、planner soft、预算 fit、用户硬预算、餐饮/景点多样性分别给分。
- `meal_cost_scale_ok` 已经较高时，更适合做 guardrail，不要继续过重加权。

## 当前推荐流程

1. 固定主目标：`planner_soft_pass`，同时把 `hard_pass`、`recomputed_budget_user_constraint_ok`、`recomputed_budget_fit_ok` 作为保护指标。
2. 用 eval 失败样本总结失败模式，但只从 train-only contexts / 候选池构造训练 pair。
3. pair selection 前做 frozen eval signature 过滤，并在 summary 里报告 overlap。
4. chosen 必须 hard pass，尽量 planner soft pass；rejected 最好 hard pass，但只在一个明确 planner soft 维度失败。
5. 数据配比保留 replay anchor，不要只喂最新 closing 数据。
6. 单生成训练继续做 checkpoint sweep，不默认 final 最好。
7. 评估时分开跑 single generation 和 rerank n4，不混用指标口径。
8. 单生成最佳先看 260519 ckpt-138；允许 rerank 时看 260521 ckpt-64 rerank n4。
9. 如果继续 DPO，先明确目标是提升单生成，还是改善多候选搜索空间。
10. 如果 DPO 收益变小，再做 Best-of-N + rule reward，之后再小规模试 GRPO / RLOO。
