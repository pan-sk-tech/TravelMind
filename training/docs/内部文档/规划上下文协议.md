# Planner 规划上下文协议

目标：让最终 Planner 模型像“业务编译器”一样，把显式、可审计的 PlannerContext 稳定转换成 `TripPlan JSON`。planner 不追求让模型凭空知道更多事实，而是尽量减少它需要猜的内容。

当前分工：

```text
前端负责收集用户意图。
后端负责结构化约束、调用工具、补齐价格 hint、生成 PlannerContext。
Planner 模型负责在候选和约束之内生成 TripPlan JSON。
训练数据和评估脚本必须复用同一份 PlannerContext 协议。
```

## 1. 设计原则

- 事实来自工具和后端规则，不让模型自由编造。
- 预算、人数、住宿晚数、价格口径必须结构化。
- SFT 学确定性业务协议，DPO 再学合法候选之间的质量偏好。
- 线上输入、SFT 输入、DPO prompt 必须同构。
- 每个规则都能被后置校验脚本验证。

## 2. 顶层结构

planner `PlannerContext` 顶层结构：

```json
{
  "version": "planner_context",
  "generated_at": "2026-05-02T15:30:00+00:00",
  "request": {},
  "party": {},
  "budget_constraint": {},
  "preference_profile": {},
  "lodging_policy": {},
  "pricing_policy": {},
  "route_policy": {},
  "tool_snapshot": {},
  "planner_constraints": {}
}
```

字段职责：

| 字段 | 主要来源 | 职责 |
| --- | --- | --- |
| `request` | 前端 planner 请求 | 保留用户原始旅行请求 |
| `party` | 前端显式输入 | 人数和同行类型 |
| `budget_constraint` | 前端显式输入 | 总预算语义 |
| `preference_profile` | 后端从标签和自由文本解析 | 正向偏好、负向约束、饮食约束、节奏 |
| `lodging_policy` | 后端规则 | 住宿晚数和酒店价格单位 |
| `pricing_policy` | 后端规则 | 酒店、景点、餐饮、交通价格口径 |
| `route_policy` | 后端规则 | 路线时间单位、路线估算来源和可接受阈值 |
| `tool_snapshot` | 工具调用、缓存、本地资产表 | 天气、景点、酒店、餐饮候选 |
| `planner_constraints` | 后端规则 | 输出天数、日期、每日项目数、grounding 规则 |

## 3. TripRequest 显式字段

planner 的前端请求 `TripRequest` 除了城市、日期、交通、住宿偏好，还必须包含结构化人数和预算：

```json
{
  "city": "北京",
  "start_date": "2026-04-17",
  "end_date": "2026-04-18",
  "travel_days": 2,
  "transportation": "自驾",
  "accommodation": "民宿",
  "preferences": ["美食", "第一次来", "博物馆"],
  "free_text_input": "和对象一起，总预算控制在1200元左右，希望节奏慢一点",
  "party": {
    "adults": 2,
    "children": 0,
    "elders": 0,
    "total": 2,
    "companion_type": "couple"
  },
  "budget_constraint": {
    "amount": 1200,
    "scope": "total",
    "currency": "CNY",
    "budget_level": "limited",
    "strictness": "soft"
  }
}
```

planner 不兼容旧请求，前端必须显式提交人数和预算结构：

- `party` 必须存在，且必须满足 `total = adults + children + elders`。
- `budget_constraint` 必须存在。
- 如果用户没有明确预算，前端仍提交 `amount=null, strictness="none"`。
- 自由文本可以保留用户原话，但不能代替结构化人数和预算字段。

进入 `PlannerContext` 后，`party` 和 `budget_constraint` 会被提升为顶层字段；`PlannerContext.request`
只保留城市、日期、交通、住宿偏好、偏好标签和自由文本，避免同一约束重复出现。

## 4. Party

`party` 表示同行人数和旅行类型：

```json
{
  "adults": 2,
  "children": 0,
  "elders": 0,
  "total": 2,
  "companion_type": "couple"
}
```

字段规则：

- `total = adults + children + elders`。
- planner 第一版预算先按 `total` 粗算，不做儿童、老人、学生、军人等优惠价细分。
- `companion_type` 可取：`solo`、`couple`、`friends`、`family_with_children`、`family_with_elders`、`business`、`other`。
- 带娃、老人、行动不便会影响每日景点数量、交通方式和酒店位置。

## 5. Budget Constraint

`budget_constraint` 统一预算语义：

```json
{
  "amount": 1200,
  "scope": "total",
  "currency": "CNY",
  "budget_level": "limited",
  "strictness": "soft",
  "source": "frontend"
}
```

字段规则：

- planner 第一版主线只训练 `scope = "total"`，即整趟旅行总预算。
- 后续可以扩展 `per_person_total`、`per_day_total`，但不进入第一轮 SFT 主线。
- `strictness` 可取 `hard`、`soft`、`none`。
- `budget_level` 可取 `limited`、`standard`、`comfortable`、`premium`、`luxury`。
- 如果用户没有显式预算，`amount=null, strictness="none"`。

预算遵守规则：

- `strictness="hard"` 时尽量不超过 `amount`。
- `strictness="soft"` 时可以轻微浮动，但必须在建议里解释。
- 没有预算时仍要给合理估算，但不做超预算惩罚。

## 6. 约束拆分

自然语言仍然保留，但训练和评估应优先使用结构化约束：

```json
{
  "preference_profile": {
    "positive_preferences": ["博物馆", "本地美食", "第一次来"],
    "negative_constraints": ["不吃辣", "避免长时间排队"],
    "pace": "慢节奏",
    "diet_positive": "本地菜",
    "diet_avoid": ["辣", "海鲜"],
    "traveler_constraints": {
      "needs_child_friendly": false,
      "needs_elder_friendly": false,
      "avoid_long_walk": true
    }
  }
}
```

原则：

- 负向约束优先级高于正向偏好。
- 忌口、过敏、不吃、避免类信息不能作为正向搜索关键词。
- 带娃、老人、行动不便等约束会影响每日景点数量、交通方式和酒店位置。

## 7. 住宿和预算策略

planner 显式给模型住宿晚数和酒店价格单位：

```json
{
  "lodging_policy": {
    "hotel_cost_unit": "per_night",
    "default_lodging_nights": 1,
    "last_day_hotel_default": null
  }
}
```

默认规则：

- N 天行程默认住宿 N-1 晚。
- `hotel.estimated_cost` 表示单晚价格。
- `budget.total_hotels` 必须覆盖住宿晚数。
- 最后一天返程时 `day.hotel = null`，不额外计算一晚。
- 如果最后一天继续入住，才把最后一天酒店计入总酒店费用。
- `day.accommodation` 始终填写用户住宿类型字符串，最后一天也不能写 `null`、`无住宿` 或 `返程`。

## 8. 价格口径

planner 把价格口径前置到候选和规则里，不让模型临场猜。

```json
{
  "currency": "CNY",
  "hotel_cost_unit": "per_night",
  "attraction_ticket_unit": "adult_ticket",
  "meal_cost_unit": "per_person_per_meal",
  "transportation_cost_unit": "party_total_per_day"
}
```

输出口径：

- `hotel.estimated_cost`：单晚价格。
- `attraction.ticket_price`：单人成人全价门票。
- `meal.estimated_cost`：单人单餐费用。
- `budget.total_attractions`：所有景点门票按 `party.total` 汇总。
- `budget.total_hotels`：所有非 null 酒店单间每晚价按 `rooms = ceil(party.total / 2)` 汇总。
- `budget.total_meals`：所有餐饮单人单餐价按 `party.total` 汇总。
- `budget.total_transportation`：全程交通按全体同行人汇总。
- `budget.total = total_attractions + total_hotels + total_meals + total_transportation`。

## 9. 酒店价格估算

高德酒店 POI 的 `cost` 经常为空，planner 先用规则估价，不把 OTA 实时房价作为主线。

酒店候选建议字段：

```json
{
  "name": "心悦民宿",
  "address": "康辛路鸿业兴园一区6号楼4单元1001室",
  "location": {"longitude": 116.305241, "latitude": 39.836034},
  "rating": "4.8",
  "cost": null,
  "estimated_cost_hint": 350,
  "cost_source": "rule_estimated"
}
```

初版规则，单位为元/晚：

| 住宿类型 | 基准估价 |
| --- | ---: |
| 经济型酒店 | 220 |
| 民宿 | 300 |
| 舒适型酒店 | 420 |
| 亲子酒店 | 520 |
| 高端酒店 | 750 |
| 豪华/奢华酒店 | 1200 |

城市系数：

| 城市类型 | 系数 |
| --- | ---: |
| 北京、上海、三亚、杭州 | 1.15 |
| 广州、深圳、成都、西安、青岛、厦门、南京、苏州 | 1.05 |
| 普通省会、热门二线 | 1.00 |
| 三线及以下、县域 | 0.85 |

计算方式：

```text
estimated_cost_hint = round_to_50(base_by_accommodation * city_factor)
```

如果高德 `cost` 可用，则可使用 `amap_cost`，并保留 `cost_source = "amap_cost"`。

## 10. 景点门票估算

景点候选来源统一先走高德 POI 召回，不因是否包含价格而跳过高德。高德景点 POI 的 `cost` 覆盖率较低，不能把门票预算完全交给模型猜。planner 第一版引入本地“热门景点票价画像表”。票价 hint 的优先级为：高德 `cost` -> 本地票价画像表 -> 规则估算。

本地票价画像表建议字段：

```json
{
  "city": "北京",
  "name": "故宫博物院",
  "aliases": ["故宫", "紫禁城"],
  "category": "历史文化景区",
  "ticket_price_profile": {
    "off_season": 40,
    "normal_season": 50,
    "peak_season": 60
  },
  "price_unit": "adult_ticket_cny",
  "ticket_price_source": "llm_estimated",
  "confidence": "medium",
  "estimated_at": "2026-05-02",
  "note": "强模型按成人全价常见区间估算，非实时票价"
}
```

当前版本约定：

- 票价口径先按成人全价估算。
- 淡季、平季、旺季票价由脚本根据日期选择，不交给 Planner 模型判断。
- `ticket_price_source` 可取 `amap_cost`、`manual_verified`、`llm_estimated`、`rule_estimated`。
- SFT 训练时模型只复制 `ticket_price_hint`，不自行估算票价。
- 学生、儿童、老人、军人等优惠票价先不进入 planner 第一版训练目标，后续可以扩展为更细的 `ticket_price_profile_by_traveler_type`。

景点候选建议字段：

```json
{
  "name": "故宫博物院",
  "address": "景山前街4号",
  "location": {"longitude": 116.397026, "latitude": 39.918058},
  "rating": "4.8",
  "ticket_price_hint": 60,
  "ticket_price_season": "peak_season",
  "ticket_price_source": "llm_estimated"
}
```

预算规则：

```text
budget.total_attractions = sum(selected_attraction.ticket_price_hint) * party.total
```

如果儿童、老人等优惠票价尚未显式提供，当前版本仍按 `party.total` 使用成人全价估算，保证预算偏保守且账本可校验。

## 11. 餐饮和交通价格

餐饮候选建议字段：

```json
{
  "name": "方砖厂69号炸酱面",
  "type": "餐饮服务;中餐厅",
  "district": "东城区",
  "address": "崇文门附近",
  "location": {"longitude": 116.421, "latitude": 39.904},
  "rating": "4.6",
  "cost": 45,
  "cost_unit": "per_person",
  "meal_cost_hint": 90,
  "cost_source": "amap_cost",
  "price_level": "standard",
  "meal_roles": ["lunch", "dinner"],
  "cuisine_tags": ["本地菜", "老字号"],
  "diet_tags": [],
  "avoid_risk_keywords": [],
  "matched_keyword": "本地菜",
  "source_bucket": "food_base"
}
```

规则：

- `food_query_groups` 记录餐饮召回分桶，例如 `food_breakfast`、`food_base`、`food_preference`、`food_diet`、`food_companion`、`food_budget`。
- `source_bucket` 表示该餐厅来自哪个召回分桶，`matched_keyword` 表示命中的搜索关键词。
- 当前主链路不对 `food_pois` 做 topk 截断；`food_base` 作为兜底池默认返回 20 个候选，`food_breakfast` 默认返回 8 个候选，其他餐饮分桶默认返回 8 个候选，合并后总上限默认 40。
- `meal_roles` 表示候选适合哪些餐别；早餐候选可用于 breakfast，普通餐厅主要用于 lunch/dinner。
- `cuisine_tags` 表示口味/场景标签，例如本地菜、老字号、小吃、咖啡、清真、素食、亲子餐厅。
- `diet_tags` 表示可正向满足的饮食标签；`avoid_risk_keywords` 表示可能触犯忌口/过敏约束的关键词，正常情况下应被过滤为空。
- `price_level` 是餐饮人均价格粗档位：`budget/standard/premium/luxury`。
- 如果餐饮 `cost` 可用，默认理解为人均价格。
- `meal_cost_hint = cost * party.total`，模型输出 `meal.estimated_cost` 时复制 `meal_cost_hint`。
- 训练数据中如果某个餐饮候选没有价格且没有后端 fallback，优先作为脏数据剔除。
- 线上系统可以用餐别 fallback：早餐 25/人，午餐 60/人，晚餐 90/人，再乘以 `party.total`。

餐饮 grounding 硬约束：

- 每天必须包含 `breakfast/lunch/dinner` 三餐；最后一天/返程日也必须安排 `dinner`，如果晚上返程则安排离站/离城前的具体晚餐餐厅。
- `lunch` 和 `dinner` 必须从 `tool_snapshot.food_pois` 中选择，并复制候选的 `name/address/location/meal_cost_hint`。
- `breakfast` 优先从 `food_breakfast` 或 `meal_roles` 含 breakfast 的 `food_pois` 中选择；只有到达日、赶早行程或亲子老人省力场景下，才可以使用 `酒店早餐`、`民宿早餐` 等住宿早餐，且整趟行程住宿早餐最多出现 2 次。
- `lunch/dinner` 绝不能为空，绝不能使用 `酒店午餐`、`酒店晚餐`、`民宿晚餐`、`客栈晚餐`、`酒店餐厅` 等住宿类餐饮，绝不能把 `hotel_pois` 中的酒店/民宿/客栈/旅舍名称写成 `meal.name`。
- 不允许输出泛化占位餐饮，例如 `早餐推荐`、`午餐推荐`、`晚餐推荐`、`附近餐厅`、`当地小吃`、`本地餐馆`、`酒店晚餐`、`无`。
- 如果当天周边没有特别匹配的餐饮候选，仍应从 `food_pois` 里选择最接近偏好、位置或预算的候选，而不是自行编造餐厅。
- `meal.estimated_cost` 必须使用候选的 `meal_cost_hint`；如果使用住宿早餐 fallback，则按后端早餐 fallback 口径估算。

交通第一版用规则估计：

- 公共交通：30 元/人/天。
- 步行：0-10 元/天。
- 打车/混合：80-150 元/天，按城市和天数估算。
- 自驾：停车和市内通行按 80-150 元/天估算，不含跨城油费。

交通费用进入 `budget.total_transportation`，不用拆到每天的 schema 里。

## 12. 路线时间

路线时间直接影响行程是否可执行，但 planner 当前主线不再使用粗糙的直线距离路线估算。

当前版本直接把酒店、景点、体验和餐饮候选的 `district`、`address`、`location` 交给 Planner，由模型自行安排顺路组合。真实路线时间增强作为后续工程项处理，避免半成品 `route_hints` 误导模型或污染训练数据。

`route_policy` 建议字段：

```json
{
  "transportation_mode": "公共交通",
  "route_hint_status": "disabled",
  "route_basis": ["poi_district", "poi_address", "poi_location"],
  "future_enhancement": "amap_route_api_with_cache"
}
```

`tool_snapshot.route_hints` 当前固定为空数组：

```json
[]
```

构造策略：

- 不计算所有候选两两路线，避免打爆 API 和上下文。
当前版本：

- 不生成路线时间 hint。
- 不使用直线距离估算作为 Planner 输入。
- Planner 只能根据候选的 `district`、`address`、`location` 自行判断动线。

后续增强：

- 只对必要候选调用高德路线 API。
- 每个城市和请求使用缓存，缓存 key 包含城市、交通方式、起终点坐标。
- 真实路线 API 结果可生成 `route_hints`，并标记 `source = "amap_route"`。

SFT 规则：

- 优先把同一天景点安排在区域相近、地址相近或经纬度相近的组合里。
- 避免同一天出现明显跨区跳跃。
- 如果不得不跨区，必须在 `description` 或 `transportation` 中解释。

## 13. 工具快照

`tool_snapshot` 是 Planner 的事实来源：

```json
{
  "classic_pois": [],
  "preference_pois": [],
  "scenic_pois": [],
  "experience_pois": [],
  "hotel_pois": [],
  "food_pois": [],
  "food_query_groups": [],
  "route_hints": [],
  "available_weather": [],
  "trip_weather": [],
  "candidate_counts": {},
  "tool_status": {}
}
```

候选分类：

- `classic_pois`：城市经典景点，用于解决只按偏好搜索遗漏主要景点的问题。
- `preference_pois`：按用户正向偏好搜索的景点。
- `scenic_pois`：经典和偏好合并后的景点候选。
- `experience_pois`：体验类、展馆类、活动类候选。
- `hotel_pois`：酒店候选，必须带 `estimated_cost_hint`。
- `food_pois`：餐饮候选，优先带 `meal_cost_hint`。
- `food_query_groups`：餐饮召回分桶和关键词，用于审计餐饮候选是否覆盖偏好、预算、同行和饮食约束。
- `route_hints`：当前为空数组；后续接入真实路线 API 后再启用。
- `trip_weather`：行程每一天的天气；远期不可查时用显式未知占位。

Planner 只能使用 `trip_weather`，不能把短期天气平移到未来日期。

## 14. 工具字段复制协议

模型选择某个工具候选后，应尽量原样复制：

- `name`
- `address`
- `location`
- `rating`
- `estimated_cost_hint -> hotel.estimated_cost`
- `cost_source`
- `ticket_price_hint -> attraction.ticket_price`
- `ticket_price_source`
- `meal_cost_hint -> meal.estimated_cost`

模型不应该同名改地址、同名改坐标、同名改价格。

## 15. SFT planner 要学习的确定性能力

SFT planner 重点学习：

- 严格输出 `TripPlan JSON`。
- 按 `party.total` 估算门票、餐饮和部分交通费用。
- 按 `budget_constraint.scope = total` 理解总预算。
- 按 `lodging_policy.default_lodging_nights` 计算酒店总费用。
- 复制工具候选字段，不自由编价、编坐标。
- 复制景点候选的 `ticket_price_hint`，并按人数汇总景点预算。
- 复制酒店候选的 `estimated_cost_hint`，并按住宿晚数汇总酒店预算。
- 复制餐饮候选的 `meal_cost_hint`，并按三餐汇总餐饮预算。
- 午餐和晚餐必须选择具体 `food_pois`，最后一天/返程日也必须安排具体晚餐；不输出泛化餐厅名、附近餐厅、酒店晚餐、酒店午餐、住宿类名称或无餐饮。
- 使用候选 POI 的区域、地址和坐标避免明显绕路和跨区跳跃。
- 严格复制 `trip_weather`。
- 区分正向偏好和负向约束。
- 最后一天轻量安排和返程住宿规则。
- 工具缺口时诚实降级。

## 16. DPO planner 要学习的偏好能力

DPO planner 再学习：

- 同样合法的计划里哪个更符合预算档位。
- 哪个酒店位置更好。
- 哪条动线更顺。
- 高预算时如何提升体验。
- 低预算时如何保住主要体验。
- 小众偏好和经典景点如何平衡。
- 亲子、老人、情侣、朋友、独行等不同同行类型下的体验取舍。
- 同样合法的计划里哪条路线更省时、更少折返。

## 17. 后置校验

planner clean SFT 数据至少要通过：

- JSON/schema/date/weather 校验。
- `party.total = adults + children + elders`。
- 中间天酒店合法。
- `hotel.estimated_cost == selected_hotel.estimated_cost_hint`。
- `attraction.ticket_price == selected_attraction.ticket_price_hint`。
- `meal.estimated_cost == selected_food.meal_cost_hint`。
- `lunch/dinner.name` 必须命中 `food_pois.name`；`breakfast.name` 必须命中 `food_pois.name` 或合法住宿早餐 fallback。
- 餐饮名不得是占位词、泛化词或空值，例如 `附近餐厅`、`当地小吃`、`酒店晚餐`、`无`。
- `budget.total_hotels` 覆盖住宿晚数。
- `budget.total_attractions` 按 `party.total` 计算。
- `budget.total` 与分项加总一致。
- 餐饮、门票等预算考虑 `party.total`。
- 负向约束不被明显违反。
- 最后一天如果 `hotel=null`，不得把“无住宿/返程”写入 `hotel.name`。
- 同一天内不应出现明显过多的长距离跳转；如有长路线，需有解释。

## 18. 前后端改造顺序

planner 是破坏式升级，不保留旧请求兼容分支。建议改造顺序：

1. 后端 schema 直接升级为 planner 请求，要求 `party` 和 `budget_constraint`。
2. 后端 `PlannerContextBuilder` 输出 `planner_context`。
3. 后端补齐 `preference_profile`、`lodging_policy`、`pricing_policy`。
4. 后端给酒店候选补 `estimated_cost_hint`。
5. 后端给景点候选补 `ticket_price_hint`。
6. 后端给餐饮候选补 `meal_cost_hint`。
7. 当前版本不补 `route_hints`，直接给 Planner 候选 POI 的区域、地址和坐标。
8. Planner prompt 按 planner 口径更新价格、人数、预算和动线规则。
9. 前端同步改成 planner 表单和 planner 请求体。
10. 线上链路 smoke 通过后，再生成 planner SFT 数据。
11. planner SFT clean/eval 稳定后，再进入 planner DPO。

## 19. 第一轮 planner 暂不做的事

这些作为后续增强，不阻塞 planner 第一轮：

- OTA 酒店实时价格。
- 学生、儿童、老人、军人等优惠门票。
- 实时景区票价和预约余票。
- 跨城大交通费用。
- 每个景点精确开放时间。
- 每段路线实时路况和精确交通费用。
