# Planner 全参微调实验备忘录

这份备忘录记录 Planner 旅行规划模型尝试全参微调时的判断标准、推荐配置和风险边界。当前主线仍然是 LoRA SFT + 规则评测 + DPO/rerank；全参微调只作为能力上限实验，不作为默认训练路线。

## 1. 为什么暂不把全参作为主线

Planner 任务难在长 `PlannerContext` 下稳定执行业务协议，不是单纯把语言风格训得更顺：

- 输出合法 `TripPlan JSON`。
- 日期、天气、住宿晚数和每天三餐完整。
- 酒店、景点、餐饮来自候选，减少编造。
- 预算档位、价格 hint 和用户约束口径稳定。
- 输出可以被后处理、预算重算和 rule eval 消费。

这些目标优先适合用 LoRA SFT 学协议，再用规则和 DPO/rerank 补质量。全参微调会更新 7B 全部参数，训练成本高，也更容易把基座模型的通用能力、格式稳健性或安全余量训歪。

## 2. 当前机器是否能玩

当前训练机器具备试验条件：

```text
GPU: 8 x NVIDIA A100-SXM4-40GB
System RAM: about 503 GiB
Base model: Qwen2.5-7B-Instruct / Qwen3-8B
```

结论：

- 可以做 7B/8B full SFT smoke。
- 可以尝试小样本、1 epoch、ZeRO-3 offload。
- 不建议一开始直接跑全量 1679 条 x 4 epoch。

仓库已有一个 Qwen3-8B full SFT smoke 参考配置：

```text
training/configs/qwen3_8b/full_sft_qwen3_8b_smoke.yaml
```

## 3. 推荐先做的 full SFT smoke

如果后续要验证 Qwen2.5-7B 全参能力上限，建议新建一个 smoke 配置，先只跑 128 或 256 条。

```yaml
### model
model_name_or_path: Qwen/Qwen2.5-7B-Instruct
trust_remote_code: true
flash_attn: auto

### method
stage: sft
do_train: true
finetuning_type: full
deepspeed: examples/deepspeed/ds_z3_offload_config.json

### dataset
dataset_dir: training/data/llamafactory
dataset: trip_planner_260509_main_clean_train
eval_dataset: trip_planner_260509_main_clean_test
template: qwen
cutoff_len: 24576
max_samples: 128
preprocessing_num_workers: 16
dataloader_num_workers: 4

### output
output_dir: training/outputs/qwen25_7b/full_sft_smoke
logging_steps: 1
save_steps: 20
save_only_model: true
report_to: none
overwrite_output_dir: true

### train
per_device_train_batch_size: 1
gradient_accumulation_steps: 4
learning_rate: 5.0e-6
num_train_epochs: 1.0
lr_scheduler_type: cosine
warmup_ratio: 0.03
bf16: true
gradient_checkpointing: true
ddp_timeout: 180000000

### eval
per_device_eval_batch_size: 1
eval_strategy: steps
eval_steps: 20
```

说明：

- `learning_rate` 先用 `5e-6`，比 LoRA 小一个数量级，更稳。
- `max_samples` 先限制到 128，确认能跑、速度和 loss 曲线正常。
- `cutoff_len` 保持 24576，避免为了实验速度改变 Planner 的长上下文前提。
- `gradient_checkpointing` 和 ZeRO-3 offload 都先打开，优先保证不 OOM。

## 4. 时间预估

粗略估计，实际会受空闲 GPU 数、ZeRO-3/offload、长上下文长度和数据长度分布影响。

```text
4 x A100 40GB:
  128 samples smoke: 30 min - 2 h
  256 samples:       1 h - 4 h
  full 1679, 1 epoch: 6 h - 12 h
  full 1679, 4 epoch: 24 h - 48 h

8 x A100 40GB:
  可能接近减半，但长上下文通信和 offload 不保证线性加速。
```

## 5. 必须观察的指标

全参 smoke 不看 train loss 一项定胜负。至少观察：

- 是否稳定跑完，不 OOM，不出现 NaN。
- `sec/step` 相比 LoRA 慢多少。
- `eval_loss` 是否明显优于同数据 LoRA。
- rule eval 是否提升：
  - `json_extract_ok`
  - `hard_pass`
  - `sft_budget_semantic_hard_pass`
  - `attraction_grounding_ok`
  - `hotel_grounding_ok`
  - `meal_grounding_ok`
  - `budget_selection_ok`
- 人工抽检是否出现过拟合、模板化、重复候选、预算/餐饮选择变怪。

## 6. Go / No-Go 标准

建议按三档决策：

```text
No-Go:
  full smoke 只是 train loss 更低，但 eval_loss/rule eval 不优于 LoRA；
  或者输出更模板化、grounding 退化、预算选择变差。

Maybe:
  eval_loss 更低，但 rule eval 基本持平。
  可以再试 512 samples 或更低 LR。

Go:
  eval_loss 明显更低，rule eval 主要指标也优于 LoRA，
  且人工抽检没有明显训歪。
  再考虑 full data 1 epoch。
```

不建议直接跑 full data 4 epoch。即使 full 1 epoch 效果好，也应先评估 checkpoint，再决定是否继续训练。

## 7. 与当前主线的关系

当前推荐优先级仍然是：

```text
1. LoRA SFT: 学 schema、grounding、字段口径和基础预算语义。
2. 规则后处理: 兜预算加总、日期、人数、房间数等确定性计算。
3. rerank / DPO: 在 hardpass 候选之间学质量偏好。
4. full SFT: 只作为能力上限实验。
```

如果 LoRA r32 + 更好数据 + DPO/rerank 已经能稳定提升业务指标，全参微调可以继续搁置。
