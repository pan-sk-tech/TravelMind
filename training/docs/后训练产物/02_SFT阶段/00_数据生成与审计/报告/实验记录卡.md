# 实验记录卡：SFT 数据生成与审计

## 一、阶段目标

这一阶段的目标是把 prompt 阶段沉淀下来的协议和评测口径，变成可训练、可审计、可复现的 SFT 数据。重点不是“凑更多样本”，而是保证训练样本符合旅行规划业务口径：城市、日期、酒店、餐饮、景点、预算字段都能被结构化验证。

## 三、数据审计关注点

SFT 数据重点审计这些维度：

- JSON 是否可解析，schema 是否稳定。
- 城市、日期、天数、day index 是否和用户请求一致。
- 酒店、餐厅、景点是否来自上下文候选。
- 酒店价格是否按房间数和晚数理解。
- 餐饮价格是否按人均单餐理解。
- 景点门票是否按单人票价和人数理解。
- 总预算、分项预算和实际选择是否能重算。
- 数据分布是否覆盖城市、预算档位、旅行天数和用户偏好。

## 四、阶段判断

这一步最重要的结论是：SFT 数据不能只看“像不像一个旅行计划”，必须能被规则重算。旅行规划任务里的预算字段、酒店晚数、餐饮人均价、景点人数关系，都需要在数据里提前定清楚，否则训练会把错误口径固化进模型。

## 五、对后续训练的影响

- Clean SFT 能稳定提升 schema、grounding、预算关系和餐饮尺度，前提是数据口径足够干净。
- Usage700 和 Best-of-N 数据的收益都建立在这一轮审计基础上。
- 后续 DPO/RL 的 chosen/rejected 也沿用了这些规则审计思路，只是目标从 supervised target 变成 pairwise preference。

## 六、复盘结论

SFT 数据生成与审计阶段的价值，不是某个具体模型分数，而是建立了后面所有训练的“度量地基”：没有这套审计，后续 SFT、rerank 和 DPO 的收益都很难解释。
