# 实验记录卡：评测集与指标

## 一、阶段目标

这一阶段的目标是先把“考试标准”固定下来，再谈 prompt、SFT、DPO 或 rerank 的效果。旅行规划不是只看 JSON 是否能解析，后续所有训练都需要同时回答三个问题：输出能不能被系统消费、选择是否来自工具候选、预算和用户偏好是否真的贴合。

因此本阶段做了两件事：

1. 定义 planner 的 hard / soft 分层指标。
2. 重建 standard / hard 评测集，修正预算口径和候选供给问题。

## 二、评测指标口径

核心指标文档：`报告/评测指标.md`

本阶段把指标拆成两层：

| 展示名 | 内部字段 | 定位 | 用途 |
| --- | --- | --- | --- |
| hardpass | `sft_hard_pass` | 硬约束 | 判断输出是否可解析、可重算、可后处理。 |
| hardpass_no_budget_sum | `sft_no_budget_sum_hard_pass` | 去预算加和硬约束 | 不因模型自报 budget 精确账本失败而一票否决。 |
| hardpass_strict | `sft_strict_hard_pass` | 严格账本硬约束 | 观察模型原生预算加和能力，不作为当前主验收目标。 |
| softpass | `dpo_soft_pass` | 软质量 | 在 hardpass 通过基础上看多样性、预算偏好和选择质量。 |
| legacy_hard_pass | `legacy_hard_pass` | 历史兼容 | 旧口径，仅用于和历史报告对齐。 |

阶段判断：SFT 主要学习协议执行能力，DPO 主要学习合法候选之间的偏好选择能力。预算精确加总和最终预算贴合应由模型选择、工程重算、repair/rerank 共同承担。

## 三、评测集重建

预算口径重建报告：`报告/新预算评估集重建与SFT预算审计.md`

重建结果：

| 数据集 | 样本数 | 失败数 | records |
| --- | ---: | ---: | --- |
| standard realistic budget | 200 | 0 | `training/data/planner/eval/records.jsonl` |
| hard realistic budget | 300 | 0 | `training/data/planner/eval_hard/records.jsonl` |

关键改动：

- 住宿按两人一间和住宿晚数计算。
- 餐饮、门票、体验按真实人数线性增长。
- 市内交通按队伍共享估算。
- hard 集餐饮候选供给恢复正常，`food_pois=0` 问题消失。
- 不再把旧 SFT 请求预算分布直接当作新预算口径下的训练入口。

## 四、关键审计结论

旧 SFT 数据预算审计显示：

| 项 | 结果 |
| --- | ---: |
| 样本数 | 1119 |
| 请求预算落在新分档范围 | 36.1% |
| teacher budget 落在请求目标区间 | 68.9% |
| 候选池高配可达请求预算 | 64.2% |
| food_pois=0 | 0 |

这说明旧 SFT 数据不是简单补一点预算样本就能修好。后续应按 realbudget-only 口径重建数据，并在每一轮训练前做预算分布审计。

## 五、阶段结论

1. `hardpass` 和 `softpass` 分层是后续全链路的核心口径。
2. 预算不能只看模型输出里的 `budget.total`，必须按候选 POI 回填价格后重算。
3. 新 standard / hard 评测集已经具备后续训练横评价值。
4. 旧 SFT 请求预算和新预算表不完全对齐，继续混用会污染判断。
5. 这一步是后续 SFT、DPO、Rerank 报告能互相比较的基础。
