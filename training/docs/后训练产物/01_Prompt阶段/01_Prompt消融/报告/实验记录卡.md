# 实验记录卡：Prompt 消融阶段

## 一、阶段目标

这一阶段的目标是确认：在不训练模型的前提下，Planner prompt 和 PlannerContext 还能把旅行规划质量推多远。核心问题不是“哪一句 prompt 最漂亮”，而是确定哪些约束适合放进 prompt，哪些问题应该交给数据、评测、后处理或后续训练。

最终结论：prompt-only 收益基本到顶，后续主线不再继续堆 prompt。冻结 prompt baseline 为 `A_C`。

```text

```

当前 prompt 入口：`backend/app/agents/prompts.py::PLANNER_AGENT_PROMPT`

## 二、实验范围

Prompt 阶段经历过多轮版本家族：

| 家族 | 目的 | 结论 |
| --- | --- | --- |
| 基础 Planner / harder / no-route | 建立结构协议、景点数量、route hints、餐饮桶 | route hints 不可靠，去掉后更稳 |
| 上下文形态 | 比较 full/compact/topk/policy_first_topk | `hard_compact` 成为早期硬约束 baseline |
| 预算算术规则 | 尝试 budget_hard/order/fewshot/symbolic/ledger | prompt 不能稳定解决长 JSON 预算回扫 |
| 预算业务口径 | 明确房间数、人均餐饮、景点单人票价 | 口径要讲清，但最终预算应重算 |
| 景点来源和去重 | 防止餐厅/酒店进景点，减少重复 | source guard 有价值，去重更适合软目标 |
| 餐饮多样性与 grounding | 平衡真实候选、午晚餐、不重复和多样性 | 多目标互相拉扯，不能全压进 prompt |
| self-check | 输出前自检 schema、预算、餐饮、重复项 | 局部收益明显，但会牺牲硬指标 |
| A/B/C/D 组合 | 最终四类规则组合消融 | `A_C` 是最合适冻结点 |

## 三、最终 A/B/C/D 口径

四个块含义：

| 块 | 含义 |
| --- | --- |
| A | 预算台账优先版，强调预算计算顺序、人数、房间数、分项加总 |
| B | 酒店住宿专治版，强调 `day.hotel` 是否代表当晚住宿 |
| C | 餐厅去重专治版，强调 lunch/dinner 必须来自 `food_pois` 并减少重复 |
| D | 最终自检修复版，输出前检查 schema、天数、餐饮、预算和重复项 |

最终重评 12 组：`baseline`、`A`、`B`、`C`、`D`、`A_C`、`A_D`、`B_C`、`B_D`、`C_D`、`A_C_D`、`B_C_D`。

关键结果：

| variant | schema_ok | budget_arithmetic | hotel_budget_covers | meal_diversity_ok | meal_grounding_ok |
| --- | ---: | ---: | ---: | ---: | ---: |
| baseline | 99.00% | 50.51% | 36.36% | 71.38% | 56.23% |
| A | 98.67% | 55.07% | 35.81% | 70.27% | 56.08% |
| D | 98.67% | 49.66% | 36.49% | 73.65% | 58.11% |
| A_C | 99.33% | 58.72% | 40.27% | 66.11% | 52.68% |
| A_D | 100.00% | 49.67% | 36.00% | 71.67% | 59.33% |

## 四、主要结论

1. `A_C` 是预算表现最好的冻结点，预算算术和酒店覆盖晚数都优于 baseline。
2. self-check 类规则有局部收益，但会牺牲 schema、预算加总、住宿类型和餐饮 grounding，不适合继续加重。
3. 餐饮 grounding 与餐饮多样性存在明显拉扯，不能全靠 prompt 同时解决。
4. 预算精确算术不是 prompt 能稳定完成的问题，后续要靠数据、重算和 DPO/RL。
5. Prompt 阶段的价值是定协议、定上下文形态、定评测口径，而不是无限调文案。

## 五、后续影响

Prompt 阶段直接影响后续 SFT/DPO 的几件事：

- 固定 PlannerContext 和结构化 JSON 协议。
- 明确酒店、餐饮、景点价格字段的业务含义。
- 把预算、餐饮多样性、景点去重等问题拆成 hard / soft / 后处理三类。
- 形成 standard / hard 评测集和可复用规则评测脚本。
