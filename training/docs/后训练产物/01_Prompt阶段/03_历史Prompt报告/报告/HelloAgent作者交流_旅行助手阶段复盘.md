# 旅行助手阶段复盘

更新时间：2026-05-04

这份材料是想和 HelloAgent 的作者交流一下，我们这几天在旅行助手项目里做了什么，以及过程中对 Agent、工具调用、后训练和评估的一些真实体会。

一句话概括：我们一开始以为主要问题是“小模型不会写旅行计划”，后来发现真正的问题更像是“Agent 系统本身还不够干净、协议不够清楚、评价标准不够细”。这几天的工作重点，就是把旅行助手从一个靠 prompt 撑着的 demo，往一个可协议化、可评估、可后训练的 Planner 系统推进。

## 1. 这个旅行助手现在怎么分工

我们没有让本地后训练模型去做所有事。它不负责查景点、不负责查天气，也不负责调用地图工具。

现在的分工大概是这样：

| 模块 | 职责 |
| --- | --- |
| 前端 | 收集城市、日期、交通、住宿、预算、同行人、自由文本偏好 |
| 后端 | 结构化约束，调用高德、天气、酒店、餐饮等工具，补齐价格 hint |
| Planner 模型 | 在候选 POI、酒店、餐饮和规则约束内生成 `TripPlan JSON` |
| 校验与评估 | 判断输出是否可解析、是否符合 schema、是否遵守预算和 grounding |

这个边界对我们很重要。因为旅行规划里有很多事实性信息，比如酒店价格、天气、景点地址，这些不应该靠模型记忆或编造。模型真正该学的是：在给定事实和约束的情况下，怎么做取舍。

## 2. 这几天最大的反转：不是模型先错，是调用链先脏了

最开始，本地 Planner 经常输出坏 JSON、半截计划、局部对象，有时还会混入 `<think>`、Markdown、上一轮生成残片。

我们第一反应是：是不是小模型格式能力太差？是不是得继续补 SFT 数据？

后来排查后端调用链，发现问题更底层：

- `SimpleAgent` 默认保留 `_history`。
- 后端的 `MultiAgentTripPlanner` 是单例，多次请求会复用同一个 Planner Agent。
- 所以第二次请求时，模型看到的不只是当前城市和当前用户需求，还会看到之前杭州、北京、南京的 prompt 和失败输出。

这就很致命。旅行规划本来就是一次性结构化生成任务，历史上下文混进来以后，模型自然容易串台、截断、输出局部对象。

修复之后，我们把 Planner 改成 stateless：每次请求都是干净输入，前后清空历史。这个改动之后，很多格式问题明显缓解。

这个经验对我们很有提醒意义：不要在调用链被污染的时候评价模型能力。否则很容易把系统问题误判成模型问题，然后去做大量无效训练。

## 3. 我们把 Planner 输入升级成了 v3 协议

之前的 Planner 输入更像是“用户请求 + 工具结果摘要 + 一大段 prompt”。这种方式能跑，但问题是很多东西靠模型猜：

- 预算是总预算还是人均预算？
- 酒店价格是一晚还是总价？
- 住几晚？
- 餐饮费用是单人还是整组？
- 哪些景点是工具候选，哪些是模型自己知道的？
- 远期天气没有数据时，模型应该写未知还是自己补？

所以我们重新设计了 `PlannerContext v3`。它把这些信息都结构化：

- `request`：用户原始请求
- `party`：成人、儿童、老人、同行类型
- `budget_constraint`：预算金额、范围、币种、严格程度
- `preference_profile`：正向偏好、负向约束、饮食偏好、节奏
- `lodging_policy`：住宿晚数、酒店价格单位
- `pricing_policy`：酒店、门票、餐饮、交通的价格口径
- `tool_snapshot`：景点、酒店、餐饮、天气候选
- `planner_constraints`：日期、天数、每日景点数、grounding 规则

我们的目标不是让模型“更懂旅行事实”，而是减少模型需要猜的东西。模型越少猜，后面的校验和训练才越靠谱。

我现在更倾向于把 Planner 模型看成一个“业务编译器”：它把显式上下文编译成最终 JSON，而不是自由写攻略。

## 4. 我们也修了几个很实际的工程坑

这部分看起来不像模型训练，但对最终效果影响很大。

第一，retry 改成干净重试。

之前 retry 会把上一次错误和坏输出拼回下一轮 prompt。短文本任务里这可能有用，但旅行 Planner 的上下文已经很长了，再把半截 JSON 塞回去，只会让输入越来越脏。现在每次 retry 都用同一个干净 `planner_query`。

第二，JSON 解析器收紧了。

旧解析器会从响应里的任意 `{` 开始尝试解析。有时顶层 JSON 已经截断了，但里面某个 `day` 或 `hotel` 对象刚好完整，解析器会误以为成功。现在只接受完整顶层 `TripPlan JSON`，并且要求有 `city/start_date/end_date/days/overall_suggestions` 等关键字段。

第三，prompt 示例格式和输出要求统一了。

旧 prompt 里有 Markdown JSON 代码块，但又要求模型不要输出 Markdown。强模型可能没事，小模型很容易模仿代码块。现在去掉代码块标记，明确第一个非空字符必须是 `{`，最后一个非空字符必须是 `}`。

第四，工具候选池也重做了。

以前景点召回太依赖用户偏好。用户说“亲子、美食、休闲”，工具可能召回一些很窄的点，反而漏掉城市经典景点。强模型为了让计划像真的旅行计划，会用常识补充宽窄巷子、武侯祠、故宫这类地方。表面看是模型不听工具，本质可能是工具没给够。

所以现在拆成了：

- `classic_pois`：城市经典候选，和用户偏好无关。
- `preference_pois`：按用户兴趣召回的候选。
- `hotel_pois`：住宿候选。
- `food_pois`：餐饮候选。
- `trip_weather`：逐日天气。

这让模型既有经典旅行骨架，也能照顾个性偏好。

## 5. 相比第一版，主要进步在哪里

第一版更像是一个能跑起来的旅行规划 Agent。现在 v3 更像是一个可以评价、可以迭代的 Planner 系统。

| 维度 | 第一版 / 早期 v2 | 当前 v3 阶段 |
| --- | --- | --- |
| 输入协议 | 工具摘要和 prompt 拼接较多，字段口径不稳定 | `PlannerContext v3` 显式描述请求、人数、预算、候选、价格口径 |
| Agent 状态 | Planner 可能跨请求保留历史 | Planner stateless，每次请求干净输入 |
| JSON 解析 | 容错过强，可能误捞半截内部对象 | 只接受完整顶层 TripPlan |
| 工具召回 | 过度依赖偏好关键词，容易漏经典景点 | 城市经典候选 + 偏好候选分层 |
| 预算口径 | 人数、晚数、价格单位容易混 | 酒店/门票/餐饮/交通价格单位结构化 |
| 评估方式 | 更偏“能否生成” | 规则硬指标 + grounding + 预算 + LLM judge |
| 后训练判断 | 倾向继续补 SFT 格式数据 | 冻结强 baseline，DPO 学偏好质量 |

几个明显变化：

1. **硬格式稳定了很多。**

   早期 `base_qwen25_7b_prompt_v2` 的评估里，91 条样本只有 16 条通过 schema，`schema_ok` 只有 17.58%。

   到 v3 no-route baseline 的老口径评估里，300 条 harder eval 上 JSON 和 schema 都接近 99%，`hard_pass` 到过 95.30%。

2. **预算和价格口径清楚了。**

   现在明确了酒店是一晚价，默认住宿晚数是 `travel_days - 1`，门票按人数乘，餐饮是整组每餐，`budget.total` 必须等于四个分项之和。

3. **工具 grounding 更可检查了。**

   景点、酒店、餐饮是不是来自候选，现在都能算命中率。模型编没编，可以被脚本抓出来。

4. **我们对 SFT 更谨慎了。**

   SFT 能学结构，也能让每天景点数更稳。但它不一定提升整体质量，甚至可能让预算判断、预算加总和推理速度变差。所以我们现在不再把“大量补格式 SFT”当主线。

5. **DPO 的目标变清楚了。**

   DPO 不应该再学坏 JSON，也不应该学带模板词的坏样本。它应该学的是：两个计划都合法时，哪个更像一个靠谱旅行顾问写的。

## 6. 我们现在怎么评价这个 Planner

我们现在把评价分成两层。

第一层是硬指标：这个 JSON 能不能被线上系统稳定消费。

第二层是软质量：这个计划像不像真的能拿给用户用。

### 6.1 基础格式与 schema

| 指标 | 含义 | 目标 |
| --- | --- | ---: |
| `api_ok` | 模型调用成功，无空响应或服务错误 | >= 98% |
| `json_extract_ok` | 能提取完整顶层 `TripPlan JSON` | >= 98% |
| `schema_ok` | 通过后端 Pydantic `TripPlan` schema | >= 98% |
| `location_object_ok` | 经纬度字段是合法对象，而不是字符串或乱写 | 100% |

### 6.2 日期、天气和行程形状

| 指标 | 含义 | 目标 |
| --- | --- | ---: |
| `date_range_ok` | 顶层城市、开始日期、结束日期与请求一致 | 100% |
| `days_len_ok` | `days` 长度等于旅行天数 | 100% |
| `day_index_ok` | `day_index` 从 0 递增 | 100% |
| `day_dates_ok` | 每天日期逐日对应请求日期 | 100% |
| `weather_match` | 每天 `weather_info` 来自 `trip_weather` | 100% |
| `attraction_count_ok` | 每天景点数在 1-3 个之间 | >= 98% |
| `meal_complete` | 每天包含 breakfast/lunch/dinner | >= 98% |

### 6.3 住宿和价格口径

| 指标 | 含义 | 目标 |
| --- | --- | ---: |
| `middle_hotel_ok` | 住宿日酒店不为空 | 100% |
| `invalid_hotel_name_ok` | 酒店名不能是“无/返程/当天返程”等占位词 | 100% |
| `hotel_budget_covers_nights` | 酒店预算覆盖应住晚数 | 100% |
| `hotel_distance_placeholder_ok` | 没有真实路线工具时，不输出伪精确“距离景点2公里” | 100% |

### 6.4 Grounding 指标

Grounding 用来判断模型是否真的使用工具候选，而不是凭空编造。

| 指标 | 含义 | 目标 |
| --- | --- | ---: |
| `attraction_grounding_rate` | 景点命中 `classic/preference/scenic/experience` 候选 | >= 98% |
| `hotel_grounding_rate` | 酒店命中 `hotel_pois` 候选 | >= 98% |
| `meal_grounding_rate` | 正餐命中 `food_pois` 候选 | >= 90% |
| `meal_specific_ok` | 餐饮不能是“午餐推荐/当地小吃/附近餐厅”等泛化词 | >= 98% |
| `meal_grounding_ok` | lunch/dinner 必须来自候选或明确合法 fallback | >= 90% |

当前最值得注意的是餐饮。旧规则没有严格卡餐饮占位，所以老 baseline 的 hard pass 偏乐观。严格口径加入后，餐饮 grounding 立刻成为主要短板。

### 6.5 预算指标

| 指标 | 含义 | 目标 |
| --- | --- | ---: |
| `budget_arithmetic_consistent` | `budget.total` 等于景点、酒店、餐饮、交通四项之和 | 100% |
| `budget_within_user_budget` | hard 预算不超过用户上限 | 100% |
| `budget_level_aligned` | 输出花费符合 limited/standard/premium 等预算档位 | >= 90% |
| `budget_preference_aligned` | 不仅不超预算，也不要高预算过度省钱 | >= 90% |
| `budget_usage_ratio` | 记录实际总价 / 用户预算，用于切片分析 | 记录 |

预算指标不只看“是否超支”。一个高预算用户被安排成穷游，也算体验差。

### 6.6 LLM-as-Judge 质量指标

规则指标只能判断“能不能用”，不能充分判断“好不好”。因此我们追加了 LLM judge，按 1-5 分评价：

| 维度 | 含义 |
| --- | --- |
| `preference_satisfaction` | 是否满足兴趣、同行类型、节奏和自由文本偏好 |
| `practicality` | 路线、密度、酒店位置、交通安排是否现实 |
| `grounding_faithfulness` | 是否忠实使用候选和工具事实 |
| `budget_reasonableness` | 钱是否花得合理，不乱超也不过度省 |
| `coherence` | 每天安排是否连贯，没有重复、矛盾或返程日逻辑错 |
| `overall_quality` | 综合判断是否像真实旅行顾问给出的计划 |

### 6.7 DPO 阶段相对指标

DPO 不主要学习坏 JSON 或坏 schema。候选必须先通过硬过滤，再比较谁更好。

| 指标 | 含义 | 目标 |
| --- | --- | ---: |
| `hard_pass` | DPO 输出不能牺牲基础协议稳定性 | 不低于 baseline |
| `win_rate_vs_base` | 同一上下文下，DPO 相比 base 的 pairwise 胜率 | > 55% |
| `judge_overall` | LLM judge 综合均分 | 高于 base |
| `budget_quality` | 预算花费结构更合理 | 高于 base |
| `preference_alignment` | 偏好和负向约束权衡更自然 | 高于 base |
| `feasibility` | 路线、酒店位置、日程密度更可执行 | 高于 base |
| `weather_awareness` | 雨天、高温、远期未知天气处理更合理 | 高于 base |
| `pair_valid_rate` | 进入 DPO pair 的 chosen/rejected 都通过硬过滤 | 100% |

## 7. 当前结果，好的和不好的都说

### 7.1 老口径下，硬协议已经比较稳

当前曾冻结 `base_qwen25_7b_v3_harder_no_route_w8` 作为 v3 prompt baseline。该版本去掉了粗糙直线距离 `route_hints`，只给模型候选 POI 的区域、地址和坐标。

300 条 harder eval 老口径规则结果：

| 指标 | 结果 |
| --- | ---: |
| API 调用成功 | 300/300 |
| JSON 可解析 | 99.33% |
| Schema 通过 | 99.33% |
| hard_pass | 95.30% |
| attraction_count_ok | 100.00% |
| budget_arithmetic_consistent | 96.64% |
| budget_preference_aligned | 96.31% |
| budget_within_user_budget | 100.00% |
| hotel_budget_covers_nights | 99.33% |
| weather_match | 99.66% |

这说明 base + v3 prompt 已经能稳定执行大部分确定性业务协议。

### 7.2 但 LLM judge 显示，计划质量还不够好

同一版 no-route baseline 追加 LLM judge 后，结论没有规则指标那么乐观。

| 维度 | 均分 / 5 |
| --- | ---: |
| 工具忠实 | 3.4099 |
| 偏好满足 | 2.4982 |
| 连贯性 | 2.3039 |
| 可执行性 | 2.1625 |
| 综合质量 | 1.8834 |
| 预算合理 | 1.5442 |

低分样本主要集中在：

| 问题类别 | 命中次数 |
| --- | ---: |
| 餐饮/美食 | 443 |
| 偏好/负向约束 | 424 |
| 预算/费用 | 348 |
| 酒店/住宿 | 328 |
| 交通/路线/位置 | 317 |

这说明当前系统已经比较像“合格 JSON 生成器”，但还不像一个真正靠谱的旅行顾问。

### 7.3 严格餐饮口径一加，问题立刻暴露

后来我们人工看了一些高分样本，发现老规则漏掉了餐饮占位问题。模型会写：

- 午餐推荐
- 附近餐厅
- 酒店晚餐
- 当地小吃
- 无

这些在 JSON 上是合法的，但用户看到会觉得很敷衍。

于是我们补了 `meal_specific_ok`、`meal_grounding_ok`，并把它们纳入 hard pass。用修正后的口径重新评估 `base_qwen25_7b_v3_harder_no_route_meal_hotel_prompt_w8`，结果是：

| 指标 | 结果 |
| --- | ---: |
| JSON 可解析 | 99.00% |
| Schema 通过 | 97.00% |
| hard_pass | 53.61% |
| attraction_count_ok | 100.00% |
| budget_arithmetic_consistent | 96.22% |
| budget_within_user_budget | 100.00% |
| hotel_budget_covers_nights | 98.63% |
| hotel_distance_placeholder_ok | 100.00% |
| meal_complete | 98.63% |
| meal_specific_ok | 96.91% |
| meal_grounding_ok | 58.42% |
| weather_match | 99.66% |

这个结果不是说 v3 退步了，而是说明我们的评价变诚实了。餐饮 grounding 是现在非常明确的短板。

## 8. 现在还不够好的地方

### 8.1 餐饮还不像真实推荐

旅行计划里，餐饮很影响用户感受。一个计划如果景点很完整，但午餐写“附近餐厅”，用户会立刻觉得这是模板。

所以餐饮不能只靠 DPO 学，前面工具召回、prompt、硬校验都要一起修。

### 8.2 路线还不够真实

我们去掉了粗糙 `route_hints`，因为直线距离估算会误导模型。但现在也还没有接真实路线 API。

所以模型只能靠区县、地址和坐标粗略判断，还是会出现：

- 同一天跨区跳跃。
- 酒店离主要活动区太远。
- 老人和亲子场景安排过密。
- 返程日不够轻量。

后面如果真要提升路线质量，最好接高德或地图路线 API，并做缓存。

### 8.3 预算会守底线，但不会花得漂亮

现在“不超预算”做得还行，但“预算合理”很差。

比如高预算用户可能被安排得太省，低预算用户可能酒店和餐饮结构不合理。预算总价闭合这种问题可以后置重算，但预算怎么花得像一个人类顾问，就更适合 DPO 和 judge 来优化。

### 8.4 负向约束优先级还要加强

用户说“海鲜过敏”，就不能推荐海鲜。用户说“老人少走路”，就不能安排高强度大景区。用户说“不想购物”，商业街就不能硬塞。

这类约束应该高于正向偏好。模型现在还不够稳定。

### 8.5 长上下文和长输出仍然贵

旅行 Planner 的输入很长，输出也很长。SFT 和 DPO 都会吃上下文长度。我们已经做了 compact context、动态 token 上限，也改了 DPO logprob 的分块计算，但性能仍然是长期问题。

这里的经验是：长上下文任务的成本不只是参数量，prefill、KV cache、DPO 的 chosen/rejected logprob 都会放大显存和时间。

## 9. 我们现在对 SFT 和 DPO 的判断

SFT 的价值是让模型学会“怎么答”：字段结构、JSON、每天几项、酒店和天气怎么放。

但 SFT 不一定能让模型学会“怎么选”：预算怎么花、路线怎么顺、老人亲子怎么安排、偏好和禁忌怎么权衡。

我们之前的 SFT 实验也看到类似现象：结构有所提升，但预算相关指标可能退化，输出更长更慢。

所以现在的路线是：

- SFT 作为协议 warm-up，不盲目扩大。
- Base + v3 prompt 作为强 baseline。
- DPO 用来学合法候选之间的偏好质量。

DPO 数据也要换思路。早期那种“把 chosen 手工改坏”的 rejected 很容易泄漏模板特征。后面应该用自然候选：

```text
同一个 PlannerContext
  -> base 低温生成
  -> base 高温生成
  -> strong model 低温生成
  -> strong model 高温生成
  -> 先过硬过滤
  -> judge 评分
  -> chosen = 高分合法候选
  -> rejected = 低分但自然合法候选
```

这样模型才有机会学到真正的旅行规划偏好，而不是学会识别“本方案存在问题”这种提示词。

## 10. 这几天最重要的经验

1. **不要在调用链污染时评价模型能力。**

   本地 Planner 早期坏 JSON 的核心原因不是模型不行，而是历史上下文串台。

2. **结构化长输出任务里，retry 不一定要 repair。**

   把坏输出塞回 prompt，可能让长上下文越来越脏。干净重试更稳。

3. **JSON 解析器不能太聪明。**

   宁可明确失败，也不要把半截响应里的内部对象误当成成功。

4. **SFT 不是万能补丁。**

   SFT 能学结构，但可能损伤基模已有的预算、计算和泛化能力。

5. **DPO rejected 不能带模板泄漏。**

   如果 rejected 明写“本方案存在偏好问题”，模型学到的是模板词，不是旅行偏好。

6. **半成品工具信号会误导模型。**

   粗糙直线距离 `route_hints` 不如不给。路线增强应该接真实路线 API。

7. **工具候选池质量决定训练数据质量。**

   如果候选漏掉城市经典景点，强模型用常识补齐不一定是错，应该先修召回。

最后总结一下我们现在的理解：

```text
工具给事实，
后端给约束，
模型做取舍，
规则守底线，
judge 评体验。
```

旅行助手不是单纯把模型换成本地模型就完事。真正困难的是把 Agent 系统里的事实、状态、工具、协议、训练数据和评价指标都对齐。我们这几天的主要进展，就是把这些东西一点点拧到同一条线上。
