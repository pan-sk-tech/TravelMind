# 后训练踩坑与经验总结

更新时间：2026-05-14

这份文档只保留会影响技术判断的大坑。端口、tmux、临时日志、某次服务是否挂掉，这些不放在这里。完整实验路径看 [SFT阶段总结.md](../内部文档/SFT阶段总结.md)，指标定义看 [评测指标.md](../内部文档/评测指标.md)。

这版不是流水账。它更像我给后来的自己留的提醒：哪些地方最容易误判，哪些事下次一开始就该做对。

## 先记住三句话

- 先怀疑链路，再怀疑模型。
- 先写清协议，再开始造数据。
- 指标脚本变了，历史报告必须重算。

## 1. 调用链不干净时，训练结论不可信

最重要的一次反转是：Planner 输出坏 JSON，并不主要是模型能力问题，而是调用链把输入污染了。

当时的现象很吓人：模型会只输出某一天、某个酒店对象，或者混入 Markdown、`<think>`、半截 JSON。重试几次以后仍然失败，看起来像小模型完全不会结构化输出。

后来才发现，后端复用了带 `_history` 的 Agent。第二次请求开始，模型实际看到的是当前任务加上之前城市的 prompt、失败输出和坏格式。旅行规划这种一次性结构化生成任务，不能靠隐式对话历史。

修法很朴素：

- Planner、景点、天气、酒店 Agent 都按 stateless 方式调用。
- retry 使用同一份干净 `planner_query`，失败内容只落日志，不塞回 prompt。
- JSON 解析器只接受完整顶层 `TripPlan`，不再从半截响应里捞内部对象。
- system prompt 里的 schema 示例去掉代码块标记，避免一边展示 Markdown，一边要求模型不要输出 Markdown。
- 输出长度、温度、timeout 全部显式设置。

这件事给我的教训很直接：如果输入链路是脏的，继续补 SFT 数据就是在修错方向。

## 2. SFT 解决的是协议习惯，不是所有质量问题

SFT 很有用，但它不是万能补丁。它最适合教模型这些东西：

- 按固定 schema 输出完整 `TripPlan`。
- 复制 `PlannerContext` 里的城市、日期、天气、酒店、餐饮和价格 hint。
- 让预算账本、人数关系、住宿晚数这类硬规则更稳定。
- 减少 Markdown、解释文字、局部 JSON 这类格式问题。

它不擅长替我们解决这些问题：

- 上游候选池漏掉经典景点。
- 预算口径本身没写清楚。
- 餐饮、路线、节奏、偏好取舍这类软质量排序。
- 评估规则把硬错误和软偏好混在一起。

训练 loss 也不能单独证明模型变好了。teacher-forcing loss 低，只说明模型能拟合 teacher 文本；真实生成时是否可解析、是否按预算、是否符合偏好，还要靠固定 eval 集和规则评估重新测。

现在 SFT 阶段已经收束。后面不该继续无脑加轮次，除非出现新的协议字段，或者收集到明确的新失败类型。当前更值的方向是 rerank、规则化预算修正、候选质量增强，以及少量真正偏好型 DPO。

## 3. 协议含糊时，数据越多越危险

早期很多问题看起来像模型不聪明，其实是业务协议没写出来。

典型例子：

- 用户说“总预算”，模型可能理解成人均预算或每日预算。
- 多日行程到底住几晚，没有明确 `lodging_nights`。
- 高德酒店没有稳定房价，模型会给同一家酒店编出不同价格。
- 景点门票、餐饮价格有时为空，模型只能猜。
- 请求里没有显式人数，预算关系无法验证。
- route hint 看起来像工具事实，但只是粗糙距离估算。

这些东西不能靠更多 SFT 硬补。协议没写清时，数据越多，模型越会学习强模型的猜法，而不是学习确定规则。

后来统一到这条链路：

```text
TripRequest -> PlannerContext -> compact PlannerContext -> TripPlan
```

`PlannerContext` 里显式放 `party`、`budget_constraint`、住宿晚数、价格来源、天气来源、候选来源和模型可见规则。旧口径数据能归档就归档，不能继续当主训练入口。

## 4. 工具候选池比 prompt 更早决定上限

工具增强 Planner 的质量，很大一部分取决于候选池。这个坑一开始很容易看错。

比如模型输出“武侯祠”“锦里”“宽窄巷子”，规则提示它们没有命中工具候选。第一反应会是：模型又没听工具。但人工看样本后发现，这些都是同城真实经典景点。真正的问题是第一版召回太依赖用户偏好关键词，经典景点没有稳定进入候选。

后面把景点拆成几类：

- `classic_pois`：城市经典候选，不依赖用户偏好。
- `preference_pois`：按用户兴趣召回。
- `experience_pois`：亲子、文化、休闲等体验类候选。
- `food_pois`：餐饮候选，单独做 grounding。

还有几个具体教训：

- 高德 `cost` 对景点不一定等于门票，公园、街区、夜市这类要先按规则估价。
- 宽松 POI 匹配必须精确优先、长名称优先，否则“滇池海埂公园”可能被错配到“滇池”。
- 没有真实路线 API 时，半成品 `route_hints` 可能比不给更危险。
- 没有真实距离工具时，`hotel.distance` 宁愿为空，也不要让模型写“距离景点2公里”这种伪事实。
- 天气要标清来源。历史天气可以离线增强，远期天气不可用就明确写不可用，不能逼模型编温度。

prompt 可以约束模型，但它不能弥补一个坏候选池。

## 5. 预算是账本，不是文学描述

预算问题不只是“超不超预算”。对这个项目来说，预算至少有三层：

- 字面合法：JSON 数值字段必须是最终整数，不能写 `120 + 80` 这种算式。
- 账本闭合：`total` 要等于景点、酒店、餐饮、交通分项之和。
- 语义合理：人数、天数、住宿晚数、预算档位和用户约束要互相对得上。

早期模型会在 JSON 里写算式，或者预算总额和分项加总对不上。这类问题不能放进 DPO 当 rejected。它们是硬错误，应该先被 SFT、规则校验或后置重算处理掉。

后面评估里才把预算拆开看：

- 原始预算是否自洽。
- 重算预算是否通过。
- 预算偏好是否贴合用户档位。
- 预算关系是否体现人数和天数。
- 餐饮价格尺度是否正常。

这比一个总分有用得多。总分太粗，会把真正的问题藏起来。

## 6. 餐饮和酒店最容易用占位词骗过规则

曾经有一批结果结构上很好看：每天都有 breakfast、lunch、dinner。但名字是“早餐推荐”“午餐推荐”“特色餐厅”。

旧规则只检查三餐类型是否齐全，没有检查餐厅是否具体、是否来自 `food_pois`。模型就用漂亮 JSON 包了一层空内容。

酒店也类似。`hotel.name` 可能命中候选，但 `distance` 字段照抄 prompt 示例里的“距离景点2公里”。这比空字段更糟，因为用户会以为这是工具算出来的真实距离。

后面做了两件事：

- 餐饮占位词变成硬错误，lunch/dinner 优先要求命中 `food_pois`。
- 没有真实工具支持的酒店距离默认留空，伪精确距离不再过关。

结构完整不代表内容有效。工具型任务里，grounding 是硬要求，不是加分项。

## 7. 评估指标要按训练阶段拆开

有一段时间我们把太多东西塞进一个 `hard_pass`：schema、日期、天气、酒店、预算、餐饮多样性、餐饮 grounding。结果很难判断模型到底哪里变好了。

后来拆成两层：

- `sft_hard_pass`：SFT 阶段应该守住的确定性约束。
- `dpo_soft_pass`：硬约束通过以后，再看偏好质量、餐饮多样性、预算舒适度等软指标。

这个拆分救了不少判断。否则一个模型可能 schema 很稳，只是餐饮重复，就被打成 hard fail；另一个模型餐饮更花，但预算算术掉了，看起来反而更好。

还有一个坑：评估脚本改了，旧报告就不能直接横向比。早期报告缺少新字段，如果按新口径汇总，缺失字段会被误算成 0。做横向对比前，必须用同一版 `eval_rule_metrics.py` 重跑旧 generations。

LLM Judge 也一样。judge 自己在高并发下会失败，不能把 `judge_failed` 算成被评模型 0 分。评估器失败、模型输出失败、真实低质量输出，要分开记录。

## 8. 长上下文成本不是 LoRA 能抹掉的

旅行规划天然是长上下文任务。用户请求、工具候选、天气、酒店、餐饮、预算约束全塞进去，训练和推理成本都很实在。

几个踩过的坑：

- LoRA 减少可训练参数，但不会让长上下文训练免费。
- 本地服务空载显存不高，不代表长 prompt 加并发不会炸。
- `max_tokens` 不设上限，会把 timeout 和显存风险放大。
- DPO 比 SFT 更吃显存，因为同一个 prompt 要同时处理 chosen 和 rejected。
- 某些 DPO OOM 不是卡数不够，而是实现里一次性 materialize 了巨大的 `logits.log_softmax(-1)`。

DPO 那次的修法是把目标 token log probability 改成等价的分块计算：

```text
log p(y_t) = logit(y_t) - logsumexp(logits)
```

不需要保存全词表每个 token 的 log probability，就不要保存。这个坑提醒我：显存问题不一定只能靠砍数据或加卡，有时要改中间张量的计算方式。

强模型造数也要记账。只记录字符数不够，必须保存 provider 返回的 `response.usage`，至少包括 prompt、completion、total 和 reasoning tokens。没有 usage ledger 的后台造数任务，后面根本说不清成本。

## 9. 合成数据要有账本和刹车

合成数据最怕“跑完再说”。这次项目里真正应该保留下来的做法有这些：

- 每条样本保留 `control_spec`，记录 companion、city tier、budget、diet、weather bucket 等控制变量。
- 每轮生成写 manifest：命令、模型、workers、目标成功数、输出目录、开始时间。
- 先 smoke 20，再 100，再扩大。每一步都审计 clean 率、错误分布和 token 成本。
- 外部工具调用必须有缓存，尤其是城市经典景点、酒店、餐饮和天气。
- 高德 QPS、强模型风控、失败重试，都要当成数据工程问题处理。

还有一个容易误判的点：单条样本 clean，不代表整批数据还能用。

旧数据里确实有一部分能通过新版预算审计，但它们来自旧预算生成口径。单条看起来没错，不等于整批分布符合当前训练目标。最后这些旧数据该归档就归档，不能因为“还能过几条规则”就继续混进主训练池。

## 10. 报告和路径也是实验的一部分

早期 eval 输出混在 `training/outputs/eval` 第一层，模型 run、comparison、audit、log 全堆一起。短期方便，长期很痛苦。

后来按用途拆开：

```text
training/outputs/eval/by_model/<model>/<YYMMDD>_<run_slug>/
training/outputs/eval/comparisons/<YYMMDD>_<comparison_slug>/
training/outputs/eval/audits/<YYMMDD>_<audit_slug>/
training/outputs/eval/logs/<YYMMDD>_<log_slug>/
training/outputs/eval/reports/<YYMMDD>_<report_slug>/
```

公开结论只放轻量报告，大体积 generations、日志和模型输出明细留在本地 ignored 目录。迁移清单放在 `runs_manifest.json`。

这不是洁癖。作为个人项目，最有含金量的东西就是实验记录和可复现路径。如果别人打开仓库以后找不到“哪个结论来自哪次 run”，再好的指标也会打折。

## 11. 产品壳子不是装饰

前端风格不一致这件事本身不属于算法，但它会影响项目可信度。后训练项目最后要被人看见，不是只在终端里跑通。

这次把表单、结果页、天气卡片、交通选项和偏好标签都对齐到当前 Planner 协议。这样做不是为了好看而好看，而是让线上输入、训练协议、评估报告和用户界面讲同一个故事。

## 最后判断

这个阶段最值钱的经验不是“某个 checkpoint 最强”，而是把一个旅行规划模型拆成了几类可验证问题：

- 协议是否清楚。
- 工具候选是否可信。
- 输出是否可解析。
- 预算是否能重算。
- 餐饮、酒店、天气是否 grounding。
- 评估指标是否服务当前训练阶段。

所以 SFT 到这里可以收束。后面继续做，也不应该再靠堆 SFT 数据硬冲。更合理的路线是：

- 推理时多候选生成加 rerank。
- 预算账本做规则化重算和修正。
- 路线、酒店距离、景点票价继续补真实工具。
- DPO 只处理“合法但体验差”的偏好差异。

一句话：先把系统做成干净的实验仪器，再讨论模型聪不聪明。这个顺序不能反。
