## 6. JSON / Schema 失败样例

JSON 提取失败并不都是“没输出 JSON”。这一批里很典型的原因是模型在 JSON 数值字段里写了算式，导致文本看起来像 JSON，但其实不是合法 JSON：

```json
"total_attractions": 120 + 57 + 50 + 0,
"total_hotels": 450 * 4,
"total_meals": (548 + 420 + 328) / 4 * 4
```

涉及样本：

- `v3_harder_eval_000010` `multi_party_constraint`：budget 字段内出现 `120 + 57 + ...`、`450 * 4`。
- `v3_harder_eval_000220` `multi_party_constraint`：budget 字段内出现 `180 * 5`、`400 * 4`、`310 * 3 * 5`。
- `v3_harder_eval_000252` `limited_diet_budget`：budget 字段内出现 `(548 + 420 + 328 + ...) / 4 * 4`。
- `v3_harder_eval_000120`、`v3_harder_eval_000140`、`v3_harder_eval_000160`：输出接近合法 JSON，但顶层对象提取仍失败，需要后续单独抽样看完整原文。

Schema 失败主要是字段类型或字段名问题：

- `v3_harder_eval_000135`：`days.3.meals.2.name` 缺失，疑似把中文键名写成了 `名称`。
- `v3_harder_eval_000162`：`meal.estimated_cost` 输出了小数，如 `196.8`、`107.2`，schema 要求整数。

## 7. 下一版建议

短期不建议为这些问题马上重新 SFT。更合理的顺序是：

1. 先修 rule eval / 后置校验对 `budget.total` 的处理：如果四个分项都可信，线上可以自动把 total 重算为分项和，并在日志中记录 `budget_total_repaired`。
2. 在 prompt 里再强化一条：最后必须先计算四个分项，再把 `budget.total` 写成四项之和，不允许加入未列出的费用。
3. DPO 阶段把预算质量作为 judge 维度，但 DPO pair 必须先通过预算算术硬过滤，避免让 DPO 学坏账本。
4. JSON/schema 的 8 个失败可以通过生成重试解决，不适合作为 DPO rejected。
5. 如果后续要做 SFT，目标也应是“预算账本闭合和长输出稳定性”这种窄补强，不是大规模重训格式。
