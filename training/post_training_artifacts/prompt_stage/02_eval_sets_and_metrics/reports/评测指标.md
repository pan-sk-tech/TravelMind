# Planner 评测指标

更新时间：2026-05-08

这份文档定义 planner 阶段旅行规划模型的评测指标。当前目标是先固定“考试标准”，再构建评测集、调 prompt、生成 SFT 数据和 DPO 数据。

分工：

```text
SFT 主要评估协议执行能力。
DPO 主要评估合法候选之间的偏好决策能力。
```

## 两层 Pass 口径

planner 后续不再把所有指标混成一个 `hard_pass`。评估报告固定输出两层 pass。对外展示名统一使用 `hardpass` / `softpass`；脚本内部字段暂时保留旧 key，避免历史产物和分析脚本断裂。

| 展示名 | 内部字段 | 定位 | 用途 |
| --- | --- | --- | --- |
| `hardpass` | `sft_hard_pass` | 硬约束 | 当前临时等同 `sft_no_budget_sum_hard_pass`，判断模型输出是否满足可解析、可重算、可后处理的确定性协议 |
| `hardpass_no_budget_sum` | `sft_no_budget_sum_hard_pass` | 去预算加和硬约束 | 不因模型上报 budget 精确账本失败一票否决，用于当前阶段看结构、grounding 和行程协议能力 |
| `hardpass_strict` | `sft_strict_hard_pass` | 严格账本硬约束 | 保留旧严格口径，用于观察模型原生预算加和能力，不作为当前主验收目标 |
| `hardpass_budget_semantic` | `sft_budget_semantic_hard_pass` | 预算语义硬约束 | 用“天数/人数/档位关系正确”替代“预算分项精确加总”的阶段性口径，用于判断模型是否先学会预算业务语义 |
| `softpass` | `dpo_soft_pass` | 软质量 | 在 `hardpass` 基础上看景点/餐饮多样性、预算偏好等质量项，用于判断偏好优化空间 |
| `legacy_hard_pass` | `legacy_hard_pass` | 历史兼容 | 旧口径，餐饮多样性混在 hard pass 里，仅用于和旧报告对齐 |

当前脚本里 `hard_pass` 默认等同于 `sft_hard_pass`，报告展示为 `hardpass`。餐饮 grounding 属于可重算/可后处理的基础来源约束，进入 hardpass；餐饮多样性、预算使用是否舒服、路线是否更顺等，更适合交给 softpass 和 LLM judge。

2026-05-06 临时口径：模型上报的 `budget` 精确账本不再进入当前 `sft_hard_pass` 一票否决，相关指标仍保留为诊断项，并由 `sft_strict_hard_pass` 记录旧严格结果。原因是预算加和、人数乘法、两人一间房间数属于确定性计算，更适合由工程后处理统一重算。当前 SFT 主目标先看 JSON/schema、日期天气、住宿连续性、POI grounding、三餐完整和非法占位等协议能力。

2026-05-06 预算闭环实验结论：二次调用模型可以作为预算修正的保守增强层，但不能让模型重写完整 `TripPlan`。实验中 full-plan rewrite 会破坏 schema、漏字段、把 `day.accommodation` 写空；不加验收门的 patch 会提高预算贴合但伤害景点/餐饮多样性。因此推荐线上采用 `patch + 工程验收 + 不合格回滚`：

```text
SFT 生成 TripPlanDraft
-> 工程侧按候选 POI 回填价格并计算 computed_budget
-> 若预算不贴合，第二次模型只输出 replace_hotel / replace_meal / replace_attraction patch
-> 工程侧应用 patch 到原计划副本
-> 重算预算和规则指标
-> 只有预算变好且 schema/grounding/多样性等已通过项不回退时接受 patch
-> 否则回滚原计划，再进入 deterministic greedy repair/rerank
```

在 A_C 300 条上，guarded patch 的效果小但安全：Base `budget_selection_ok` 从 24.00% 到 27.33%，SFT cp2 从 28.33% 到 30.33%；SFT `planner_draft_hard_pass` 保持 63.00%，没有被预算修正破坏。原始对比报告属于本地历史实验输出，公开版保留主要数字和结论。

2026-05-05 补充：预算问题拆成两层看。`sft_budget_semantic_hard_pass` 用于阶段性观察模型是否理解“天数 -> 住宿晚数”“人数 -> 门票总价”“人数/档位 -> 餐饮人均费用”。预算精确加总可以后续通过工程重算或数据清洗兜底，不再作为 prompt-only 消融的唯一目标。

### `hardpass` 组成

`hardpass` 只放模型应该学稳的确定性约束，当前内部字段为 `sft_hard_pass`：

- 顶层 JSON 可解析，TripPlan schema 通过。
- 城市、起止日期、天数、每天日期、`day_index` 正确。
- 天气日期和天气字段匹配 `trip_weather`。
- 每天 `accommodation` 固定等于用户请求中的住宿类型。
- 每天三餐完整，餐饮不是“无/推荐/酒店晚餐”等占位或非法名称。
- 餐饮 grounding 命中 `food_pois`；住宿早餐等明确合法 fallback 可接受。
- 每天景点数 1-3。
- 景点命中工具候选。
- 中间住宿日酒店不为空，酒店名合法，酒店命中工具候选。
- location 字段结构合法。
- 交通预算非负。

这些预算账本指标暂时不进入当前 `sft_hard_pass`，但继续输出为诊断项，并进入
`sft_strict_hard_pass` 观察：

- `budget.total` 等于四个分项之和。
- `budget.total_attractions` 等于所选景点成人单人票价之和乘以 `request.party.total`。
- `budget.total_meals` 等于所有三餐人均 `estimated_cost` 之和乘以 `request.party.total`。
- `strictness=hard` 时模型上报的 `budget.total` 不得超过 `request.budget_constraint.amount`。
- 酒店总价覆盖住宿晚数和“两人一间”的房间数。

### `sft_budget_semantic_hard_pass` 组成

`sft_budget_semantic_hard_pass` 复用大部分 SFT 硬约束，但把预算精确账本指标替换成预算语义关系指标：

- `hotel_budget_relation_ok`：酒店预算大致覆盖所有非 null `day.hotel.estimated_cost` 和“两人一间”的房间数，重点检查是否理解住宿晚数和房间数。
- `attraction_budget_party_relation_ok`：景点预算大致反映成人单人票价乘以 `request.party.total`，重点检查是否理解同行人数。
- `meal_cost_scale_ok`：午餐/晚餐 `meal.estimated_cost` 不明显低于预算档位对应的人均费用，重点检查是否把餐费写成过低占位价。
- `budget_relationship_ok`：上面三项同时通过。

这个口径不替代最终严格验收，而是用于判断 SFT 是否先学到了预算业务语义。

### `softpass` 组成

`softpass` 在 `hardpass` 之上加入更像“好行程”的软质量项，当前内部字段为 `dpo_soft_pass`。餐饮 grounding 已进入 `hardpass`，这里不再重复作为 softpass 增量项：

- `attraction_diversity_ok`：同一个景点名不要在同一趟行程中重复安排；暂不强制景点类别多样化，避免误伤历史文化游、博物馆游等垂直偏好。
- `meal_diversity_ok`：同一天午餐和晚餐尽量不吃同一家，跨天不要过度重复同一家；酒店早餐不计入多样性。
- `budget_preference_aligned`：预算使用符合用户预算档位和偏好，不明显过省或乱超。

这些软质量项是 DPO 数据构造、pairwise judge 和后续偏好训练的重点，不再作为 SFT 硬过滤的主门槛。

### 预算闭环指标

预算相关指标拆成三类，避免把“模型数学能力”和“最终用户看到的预算质量”混在一起：

| 指标 | 责任方 | 含义 |
| --- | --- | --- |
| `sft_strict_hard_pass` / `budget_arithmetic_consistent` | 模型诊断 | 观察模型原生预算账本能力，不作为当前主验收 |
| `recomputed_budget_fit_ok` / `budget_selection_ok` | 模型选择 + 工程重算 | 按模型选中的酒店/景点/餐饮回填价格后，预算是否贴合用户约束 |
| `budget_after_revision_ok` / `budget_repair_success` | 推理闭环/工程 repair | 经过模型 patch 或 deterministic repair 后，最终计划是否贴合预算 |

线上最终验收不能只看模型输出里的 `budget.total`。更合理的预算验收顺序是：

1. 忽略模型自报预算总额。
2. 从候选 POI 回填 `hotel.estimated_cost / ticket_price / meal.estimated_cost`。
3. 用人数、房间数、天数统一计算 `computed_budget`。
4. 用 `computed_budget` 判断 `budget_selection_ok`。
5. 若失败，再进入 guarded patch 或 deterministic repair，并记录 `budget_repair_success`。

预算档位贴合区间由 `budget_fit_policy` 给出。当前比例为：

| budget_level | 目标使用比例 |
| --- | ---: |
| limited | 72% - 100% |
| standard | 55% - 110% |
| comfortable | 65% - 110% |
| premium | 70% - 112% |

如果 `strictness=hard`，目标上限仍会被压到不超过 100%，也就是 hard budget 不能超预算。

2026-05-07 预算请求口径调整：评估集不再使用 `luxury` 档位；高预算需求统一落到
`premium`。用户“整趟总预算”不再按 `party.total` 完全线性增长，因为住宿按两人一间、
体验选择数量也不会随人数线性增加。评估集请求金额应满足候选池预算 100% 可达：
`estimated_high_budget / budget_constraint.amount >= 1.0`。

当前 `premium` 评估样本只保证候选池“预算 100% 可达”，不保证有很宽的高配选择余量。
尤其 `eval_hard` 中 premium 样本多为刚刚可达；高端候选池增强已记录到根目录

## SFT 阶段硬学习目标

SFT 阶段优先学习确定性业务协议。这些能力应作为 SFT 的主验收项，而不是推迟到 DPO 再解决：

| SFT 学习目标 | 评测定位 |
| --- | --- |
| JSON 可解析 | 硬门槛 |
| TripPlan schema 通过 | 硬门槛 |
| 日期、天数、`day_index` 正确 | 硬门槛 |
| 每天景点数 1-3 | 硬门槛 |
| 中间住宿日酒店不为空 | 硬门槛 |
| 酒店单价/门票/餐饮人均价字段口径正确 | 预算语义硬指标 |
| 能选出价格可回填的酒店/景点/餐饮 | 预算选择基础指标 |
| 酒店总价、景点总价、餐饮总价和 `budget.total` 精确加总 | 诊断指标；线上由工程重算 |
| `strictness=hard` 时重算预算不超过用户预算上限 | 预算选择/工程修正指标 |
| 景点/酒店 grounding 命中工具候选 | grounding 硬指标 |
| 餐饮 grounding 命中工具候选或使用合法 fallback | grounding 硬指标 |
| 天气匹配当天日期和工具天气字段 | 天气硬指标 |
| 明确负向约束不被违反 | 约束硬指标或 major fail |

其中“负向约束不被违反”主要指海鲜过敏、清真、不吃猪肉、避免爬山、老人少走路、不购物团等明确约束。宽泛审美类偏好，例如“更高级”“更舒服”“更有当地特色”，主要交给 DPO 和 LLM judge。

## SFT 指标总表

SFT 评测采用“绝对指标”：给定同一个 planner PlannerContext，模型输出是否稳定满足协议。SFT 不以“更会选路线/更会审美”为主目标，而是看是否减少业务硬错。

| 指标组 | 指标 | 计算方式 | SFT 目标 |
| --- | --- | --- | ---: |
| 格式 | `json_extract_ok` | 是否能提取完整顶层 TripPlan JSON | >= 98% |
| 格式 | `schema_ok` | `TripPlan(**data)` 是否通过 | >= 98% |
| 日期 | `date_range_ok` | city/start_date/end_date/days 长度是否和 request 一致 | 100% |
| 日期 | `day_index_ok` | 每天 `day_index` 是否从 0 递增 | 100% |
| 日期 | `day_date_ok` | 每天 `date` 是否等于请求日期序列 | 100% |
| 天气 | `weather_match` | `weather_info` 是否逐日复制 `trip_weather` | 100% |
| 住宿 | `accommodation_ok` | 每天 `accommodation == request.accommodation` | 100% |
| 住宿 | `lodging_nights_ok` | 住宿晚数范围内 `hotel != null` | 100% |
| 住宿 | `valid_hotel_name_ok` | hotel.name 不为“无/返程/无住宿”等占位词 | 100% |
| 行程密度 | `attraction_count_ok` | 每天景点数是否为 1-3 个 | >= 98% |
| 餐饮 | `meals_complete_ok` | 每天是否有 breakfast/lunch/dinner | >= 98% |
| Grounding | `attraction_grounding_rate` | 景点在 classic/preference/scenic/experience 中宽松命中 | >= 98% |
| Grounding | `hotel_grounding_rate` | 酒店在 hotel_pois 中宽松命中 | >= 98% |
| Grounding | `food_grounding_rate` | 正餐在 food_pois 中命中；早餐可放宽 | >= 90% |
| 价格复制 | `hotel_cost_hint_ok` | 酒店单晚价是否复制 `estimated_cost_hint` | 100% |
| 价格复制 | `ticket_price_hint_ok` | 景点票价是否复制 `ticket_price_hint` | 100% |
| 价格复制 | `meal_cost_hint_ok` | 餐饮是否使用 `meal_cost_hint` 或合法 fallback | >= 95% |
| 预算诊断 | `budget_arithmetic_consistent` | 模型自报 `budget.total` 是否精确等于四个分项之和 | 记录，当前不作主硬门槛 |
| 预算诊断 | `hotel_budget_covers_nights` | 模型自报 `total_hotels` 是否精确等于逐日酒店单间价 × 房间数 | 记录，当前不作主硬门槛 |
| 预算诊断 | `attraction_budget_consistent` | 模型自报 `total_attractions = sum(ticket_price) * party.total` | 记录，当前不作主硬门槛 |
| 预算诊断 | `meal_budget_consistent` | 模型自报 `total_meals = sum(meal.estimated_cost) * party.total` | 记录，当前不作主硬门槛 |
| 预算语义 | `hotel_budget_relation_ok` | 酒店预算是否大致反映实际住宿晚数和两人一间房间数 | >= 95% |
| 预算语义 | `attraction_budget_party_relation_ok` | 景点预算是否大致反映同行人数 | >= 95% |
| 预算语义 | `meal_cost_scale_ok` | 餐饮人均费用是否反映预算档位 | >= 90% |
| 预算语义 | `budget_relationship_ok` | 上述三类预算关系是否同时通过 | >= 90% |
| 预算选择 | `recomputed_budget_hard_ok` | 工程重算预算后，`strictness=hard` 是否未超过结构化预算上限 | 越高越好，SFT 阶段观察 |
| 预算选择 | `budget_selection_ok` | 工程重算预算后，是否落在 budget_fit_policy 的区间 | 越高越好，主要交给 repair/rerank |
| 预算闭环 | `budget_repair_success` | 模型 patch 或 deterministic repair 后是否预算贴合 | 线上目标 |
| 约束 | `negative_constraint_ok` | 明确负向约束是否未被违反 | >= 95% |
| 性能 | `avg_latency_seconds` | 平均生成耗时 | 记录，不作第一版硬门槛 |
| 性能 | `avg_output_chars` | 平均输出长度 | 记录，用于分析变慢/变长 |

SFT 报告中必须给出：

- 总体 hard pass。
- 每个硬指标的通过率。
- 相比 base 的绝对提升或退化。
- 失败类型 TopN。
- 按 city、天数、预算、同行类型、天气来源、住宿类型切片的指标。

第一版 SFT 成功的标准不是 judge 总分最高，而是：

```text
SFT 在 planner 协议执行硬指标上稳定不低于 base，并在预算、酒店、门票、日期等主要硬错上明显减少。
```

## DPO 指标总表

DPO 评测采用“相对指标”：同一个 PlannerContext 下，多个结构合法候选里，模型是否更偏向用户真正会满意的方案。

DPO 不训练坏 JSON、坏 schema、预算算错这类 rejected。候选必须先通过 SFT 硬门槛，再进入偏好比较。

| 指标组 | 指标 | 计算方式 | DPO 目标 |
| --- | --- | --- | ---: |
| 基础稳定性 | `sft_hard_pass` | DPO 输出是否仍通过 SFT 硬门槛 | 不低于 SFT |
| 软质量 | `dpo_soft_pass` | DPO 输出是否满足软质量门槛 | 高于 base/SFT |
| Pairwise | `win_rate_vs_base` | 同一 eval 上 DPO 相比 base 的 pairwise 胜率 | > 55% |
| Pairwise | `win_rate_vs_sft` | 同一 eval 上 DPO 相比 SFT 的 pairwise 胜率 | > 55% |
| Pairwise | `tie_rate` | judge 判为无明显差异的比例 | 记录 |
| Judge | `judge_overall` | LLM judge 综合分 | 高于 base/SFT |
| Judge | `budget_quality` | 花钱结构是否符合预算档位，不穷游也不乱超支 | 高于 base/SFT |
| Judge | `preference_alignment` | 正向偏好、同行类型、节奏是否更匹配 | 高于 base/SFT |
| Judge | `constraint_satisfaction` | 明确约束是否满足 | 不低于 SFT |
| Judge | `feasibility` | 路线、密度、酒店位置是否更现实 | 高于 base/SFT |
| Judge | `weather_awareness` | 雨天、高温、远期未知天气处理是否合理 | 高于 base/SFT |
| Judge | `text_quality` | 描述是否自然、具体、可给用户看 | 高于 base/SFT |
| 预算偏好 | `premium_not_too_cheap_rate` | premium 是否避免明显过省 | 高于 base/SFT |
| 预算偏好 | `limited_not_over_budget_rate` | limited/standard 是否避免明显超支 | 不低于 base/SFT |
| 约束偏好 | `negative_constraint_violation_rate` | 海鲜过敏、清真、不爬山等违反率 | 不高于 SFT |
| 数据质量 | `chosen_rejected_score_margin` | chosen 与 rejected 的 judge overall 差距 | 建议 >= 0.5 |
| 数据质量 | `pair_valid_rate` | chosen/rejected 是否都通过硬过滤 | 100% |

DPO 报告中必须给出：

- vs base、vs SFT 的 win/tie/lose。
- judge 各维度均分。
- 按预算、同行类型、天气、负向约束、城市层级切片的 win rate。
- chosen/rejected 分数差距分布。
- DPO 是否引入新的 hard fail。

第一版 DPO 成功的标准是：

```text
DPO 不牺牲 SFT 已学到的协议稳定性，并在预算质量、偏好对齐和可执行性上 pairwise 胜过 base。
```

## 评估集分层

planner 评估集分成两层，分别承担不同作用：

| 评估集 | 路径 | 样本数 | 用途 |
| --- | --- | ---: | --- |
| standard eval | `training/data/planner/eval/records.jsonl` | 200 | 看整体稳定性、协议执行、普通业务请求表现 |
| hard eval | `training/data/planner/eval_hard/records.jsonl` | 300 | 放大多人、多天、预算贴合、负向约束、饮食禁忌、路线可执行性差异 |

standard eval 保留接近真实分布；hard eval 不追求真实分布，而是作为压力测试，

## 2. 硬门槛指标

硬门槛判断输出能否被线上前后端稳定使用。任一失败都计入 `hard_fail`。

| 指标 | 判定方式 | 当前目标 |
| --- | --- | ---: |
| API 调用成功 | 无超时、无 4xx/5xx、无空响应 | >= 98% |
| 顶层 JSON 成功 | 复用 `extract_json_object()`，只接受完整顶层 `TripPlan` JSON | >= 98% |
| Pydantic schema 通过 | `TripPlan(**data)` 成功 | >= 98% |
| 基础形状通过 | city/start/end/days/weather/meals 符合 `validate_trip_plan_shape()` | >= 98% |
| 日期正确 | `days[*].date`、`day_index` 与请求日期序列一致 | 100% |
| 天气日期正确 | `weather_info[*].date` 与请求日期序列一致 | 100% |
| location 合法 | attraction/hotel/meal 的 location 若存在，必须是 `{longitude, latitude}` 对象 | 100% |
| 每天景点数 | 每天 `1 <= len(attractions) <= 3` | >= 98% |
| 三餐完整 | 每天包含 breakfast/lunch/dinner | >= 98% |
| 住宿类型字段 | 每个 `day.accommodation == request.accommodation` | 100% |
| 住宿日酒店合法 | `index < lodging_nights` 的 day 必须有真实 hotel | 100% |
| hotel.name 合法 | 不得为“无”“无住宿”“返程”“当天返程”等占位词 | 100% |

## 3. 预算和价格指标

planner 的预算指标必须严格遵守价格口径。

### 3.1 价格字段复制

| 指标 | 判定方式 | 目标 |
| --- | --- | ---: |
| 酒店单价复制 | `hotel.estimated_cost == hotel_pois.estimated_cost_hint` | 100% |
| 景点票价复制 | `attraction.ticket_price == POI.ticket_price_hint` | 100% |
| 餐饮价格复制或 fallback 合法 | `meal.estimated_cost` 使用 `food_pois.meal_cost_hint`，早餐/缺候选 fallback 需要记录来源 | >= 95% |

口径：

- `hotel.estimated_cost` 是单间每晚房价；酒店预算按两人一间估算房间数。
- `attraction.ticket_price` 是成人单人票。
- `meal.estimated_cost` 是人均单餐费用；餐饮预算按同行人数计算。

### 3.2 预算算术

| 指标 | 判定方式 | 目标 |
| --- | --- | ---: |
| 总预算加总 | `budget.total = attractions + hotels + meals + transportation` | 100% |
| 酒店总价 | `budget.total_hotels = sum(day.hotel.estimated_cost) * ceil(party.total / 2)`，只统计有 hotel 的住宿日 | 100% |
| 景点总价 | `budget.total_attractions = sum(ticket_price) * party.total` | 100% |
| 餐饮总价 | `budget.total_meals = sum(meal.estimated_cost) * party.total` | 100% |
| 交通总价 | 非负整数，符合交通方式和天数的合理范围 | >= 95% |

### 3.3 预算档位贴合

若 `PlannerContext.planner_constraints.budget_fit_policy.enabled = true`，评测这些指标：

| 指标 | 判定方式 | 目标 |
| --- | --- | ---: |
| hard 预算不超 | `strictness=hard` 时 `budget.total <= request.budget_constraint.amount` | 100% |
| soft 预算贴合 | `budget.total` 落在放宽后的 target 区间内 | >= 90% |
| 不明显过省 | premium 不输出明显低价穷游方案 | >= 90% |
| 不明显超支 | limited/standard 不明显超过用户预算 | >= 95% |

报告中记录 `budget_usage_ratio = budget.total / budget_constraint.amount`，并按 `budget_level`、`strictness` 切片。planner 评测必须优先读取 `request.party.total` 和 `request.budget_constraint`；只有兼容旧数据时才允许从自由文本弱推断人数和预算。

## 4. Grounding 指标

| 指标 | 候选来源 | 判定方式 | 目标 |
| --- | --- | --- | ---: |
| 景点 grounding | classic/preference/scenic/experience | `name_in_candidates()` 宽松匹配 | >= 98% |
| 酒店 grounding | hotel_pois | `name_in_candidates()` 宽松匹配 | >= 98% |
| 餐饮 grounding | food_pois | 正餐优先命中，早餐可放宽 | >= 90% |
| 价格 grounding | hint 字段 | 见预算和价格指标 | 100% |

离线评测需要统计命中率和未命中名称列表。经典但未命中工具候选的 POI 仍记为 grounding fail，用来推动工具召回和 classic 表完善。

## 5. 天气指标

天气必须来自 `PlannerContext.tool_snapshot.trip_weather`。

| 指标 | 判定方式 | 目标 |
| --- | --- | ---: |
| weather_info 覆盖 | 每天一条天气 | 100% |
| 天气字段一致 | day_weather/night_weather/day_temp/night_temp/wind 与 context 对齐 | 100% |
| 远期天气未知 | 远期不可用时保留“远期天气暂无准确预报”和“未知” | 100% |
| 天气意识 | 雨天/高温/低温在 description 或 suggestions 中有合理提醒 | judge 指标 |

天气字段一致是硬指标，天气意识是质量指标。

## 6. 路线和可执行性指标

当前 TripPlan schema 没有显式 route 数组，因此路线第一版先做软指标。

| 指标 | 判定方式 | 当前定位 |
| --- | --- | --- |
| route_hints 使用率 | 同一天酒店和景点组合是否能在 `route_hints` 中找到相关连接 | 软规则 |
| 长路线解释 | 若单段 `estimated_minutes` 超过阈值，description/transportation 是否解释 | 软规则 |
| 同日跨区跳跃 | 同一天景点 district 是否过度分散 | 软规则 |
| 每日总移动负担 | 根据 route_hints 粗估是否超过 `max_daily_transfer_minutes` | judge + 规则 |

## 7. 偏好和约束指标

负向约束优先级高于正向偏好。

| 指标 | 示例 | 判定方式 | 目标 |
| --- | --- | --- | ---: |
| 饮食避雷 | 海鲜过敏、不吃猪肉、清真、素食 | 关键词规则 + judge 复核 | >= 98% |
| 行动约束 | 老人少走路、避免爬山、带小孩 | 行程密度/景点类型/描述判断 | >= 95% |
| 避雷项 | 购物团、高价餐厅、排队、夜生活 | 关键词规则 + judge 复核 | >= 95% |

正向偏好主要看：

- 兴趣覆盖：历史文化、自然风光、美食、艺术、亲子等是否体现。
- 同行类型适配：couple/friends/solo/family/elder/business 的节奏和内容是否合理。
- pace 适配：慢节奏少景点，紧凑高效可增加密度。
- 住宿类型适配：accommodation 与 hotel type/价格档位是否一致。

## 8. LLM Judge 指标

规则指标判断“能不能用”，judge 判断“好不好”。

每个维度 1-5 分：

| 维度 | 含义 |
| --- | --- |
| 约束满足 | 是否满足预算、人数、住宿、饮食、避雷和天气硬约束 |
| 偏好对齐 | 是否体现正向偏好、同行类型和节奏 |
| 预算质量 | 是否既不超支，也不过度省钱，花钱结构是否合理 |
| 可执行性 | 路线、景点密度、酒店位置和每天节奏是否现实 |
| 工具忠实度 | 是否主要使用 PlannerContext 候选和事实 |
| 文本可用性 | 描述是否具体、自然、可直接给用户看 |
| 综合质量 | 整体旅行计划是否适合作为最终答案 |

DPO 阶段还需要 pairwise judge：同一个 PlannerContext 下，A/B 两个合法 TripPlan 哪个更好。

## 9. 切片指标

所有总指标都必须按切片展开：

| 切片 | 取值 |
| --- | --- |
| city_tier | major / popular / long_tail |
| travel_days | 2 / 3 / 4 / 5 |
| companion_type | solo / couple / friends / family_with_children / family_with_elders / business / other |
| budget_level | limited / standard / comfortable / premium |
| strictness | hard / soft / none |
| weather_provider | amap_forecast / open_meteo_archive / future_unknown |
| accommodation | 经济型酒店 / 民宿 / 舒适型酒店 / 高端酒店 / 亲子酒店 |
| transportation | 公共交通 / 地铁+步行 / 打车 / 自驾 |
| negative_constraints | 饮食避雷 / 少走路 / 不爬山 / 不排队 / 不购物团 |
| context_length | prompt_chars 分桶 |

每个切片至少报告样本数、hard_pass、budget_pass、grounding_pass、judge_overall、平均延迟、主要失败类型 TopN。

## 13. 暂不作为硬指标的内容

这些内容重要，但 planner 第一版先不做一票否决：

- OTA 实时酒店价格准确性
- 景区真实实时票价准确性
- 精确路线时长和交通换乘方案
- 餐厅实时排队、营业状态和预约难度
- 法定节假日人流预测
- 图片 URL 覆盖率

这些进入 TODO 或后续业务增强，不阻塞 planner 后训练闭环。

## 14. 当前建议执行顺序

1. 已冻结当前 planner prompt baseline：`base_qwen25_7b_planner_hard_attr_prompt_w8`。
2. 构建更难的 `eval_hard`，重点覆盖预算冲突、负向约束、天气压力、动线可执行性和多人同行成本。
3. 用当前 baseline 跑 `eval_hard`。
4. 如果 hard 指标仍稳定，则不再优先大规模补 SFT，直接转入 DPO 数据构造。
5. DPO 用多候选自然样本和 judge 学预算质量、偏好对齐、路线舒适度和文本可用性。
6. 只有当 `eval_hard` 暴露明确协议硬错时，再回到 SFT 数据增强。

```text
planner 评测先证明模型不犯业务硬错，再比较哪个合法方案更好。
```
