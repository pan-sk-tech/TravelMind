# V3 260509 Main Clean 与历史版本同口径评测对比

- 当前版本：`sft_qwen25_7b_v3_260509_main_clean_current_*_w10`
- 历史版本：`base`、`cp2_v1`、`v2a`、`v2b`
- 标准集：`training/data/v3/eval/records.jsonl`，200 条
- 困难集：`training/data/v3/eval_hard/records.jsonl`，300 条
- 推理设置：vLLM，单卡服务，`workers=10`
- 规则口径：`meal_cost_scale_ok` 只检查 lunch/dinner，早餐已排除。
- 历史数据源：`training/outputs/eval/by_model/*/260507_realbudget_*_w10/rule_eval_report.json`
- 当前数据源：
  - `training/outputs/eval/sft_qwen25_7b_v3_260509_main_clean_current_standard_w10/rule_eval_report.json`
  - `training/outputs/eval/sft_qwen25_7b_v3_260509_main_clean_current_hard_w10/rule_eval_report.json`

## 结论

260509_main 当前输出的 record_id 是 `v3_standard200_realbudget_eval_*` 和 `v3_hard_realbudget_eval_*`，所以历史基线应使用 by_model 里的 `realbudget_standard/hard` 报告。旧的 `260507_current_hard_w10` 虽然路径字符串也指向 `training/data/v3/eval_hard/records.jsonl`，但实际生成 ID 是 `v3_harder_eval_*`，不是同一批样本，不能直接用于归因。

按 realbudget 同口径看，260509_main 在困难集上的硬通过、预算关系、酒店预算、景点预算/人数、餐饮尺度和 grounding 都高于 v2b。排除早餐后，困难集餐饮尺度从 v2b 的 63.4% 提升到 70.2%。

但它不是全面胜利。相比 v2b，260509_main 的软通过、重算软通过、预算贴合、重算硬约束、预算算术和餐饮多样性下降。下一轮更值得补的是预算贴合/重算预算与餐饮多样性，而不是只盯餐饮尺度。

生成层面没有截断问题：标准集 `200/200` 都是 `finish_reason=stop`，困难集 `300/300` 也都是 `finish_reason=stop`。

1. 排除早餐后，260509_main 的困难集餐饮尺度为 70.2%，比 v2b 的 63.4% 高 6.8pp；标准集为 76.5%，比 v2b 的 71.4% 高 5.1pp。

3. 下一轮更应该补“预算贴合/重算预算”和“餐饮多样性”，餐饮尺度只需要补 lunch/dinner 低价小吃误选的问题。
